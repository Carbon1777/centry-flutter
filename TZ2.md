Нельзя сразу писать код “наугад”.
Сначала обязательна инспекция проекта, схемы БД, текущих миграций, сидов, функций, клиентских экранов и любых area/city resolver’ов.
Нужно сначала понять, что именно означает “район” в текущей системе:

ЖЁСТКИЕ ОГРАНИЧЕНИЯ

1. Не ломать существующие Москву и Санкт-Петербург.
2. Не менять без крайней необходимости существующий контракт `core_places`.
3. Новые города и районы должны быть заведены в том же стиле и через те же сущности, которые уже используются для Москвы и Санкт-Петербурга.
4. Если в проекте уже есть canonical flow для городов/районов/мест — использовать именно его.
5. Если в проекте есть area resolver, place import flow, enrichment flow, photo flow, category links — всё должно работать через уже существующий канон, а не через отдельный параллельный путь.
6. Нельзя придумывать районы вручную. Нужно брать актуальные официальные районы/эквивалентные административные единицы из авторитетных источников.
7. Нельзя молча подменять уровень административного деления. Если в городе нет точного аналога “района”, нужно сначала выяснить, какой уровень используется в текущем проекте и какой уровень будет продуктово совместим.
8. Все изменения должны быть идемпотентными: повторный запуск не должен плодить дубли и ломать данные.
9. Если в проекте есть готовые geocoded CSV с местами для этих городов, использовать их как источник для импорта мест после того, как города и районы заведены.

ОЖИДАЕМЫЙ ПЛАН РАБОТ

Этап 1. Исследование текущей модели
Нужно зафиксировать:
- какая именно модель городов/районов сейчас в проекте
- какой уровень “района” реально нужен системе
- какие данные обязательны для city
- какие данные обязательны для area
- какой формат геоданных нужен area
- как place привязывается к area

Результат этапа:
- краткий технический отчёт с реальной текущей моделью проекта
- список сущностей/файлов/миграций/функций, которые участвуют в city/area flow

Этап 2. Определение актуальных районов новых городов
Для каждого нового города нужно найти актуальный официальный список районов или эквивалентных административных единиц, которые соответствуют текущей модели проекта.

Новые города:
- Казань
- Нижний Новгород
- Краснодар
- Ростов-на-Дону
- Новосибирск
- Сочи

Правила:
- использовать официальные или максимально авторитетные источники
- не брать мусорные справочники
- не использовать устаревшие списки
- если официальный административный уровень не 1:1 совпадает с тем, как устроены areas в проекте, нужно выбрать продуктово совместимый слой и явно это обосновать

Для каждого города нужно определить:
- canonical city name
- canonical district/area names
- альтернативные формы написания, если они реально встречаются в адресах
- сколько районов / areas в итоге нужно завести
- какой уровень деления выбран и почему

Результат этапа:
- для каждого города финальный approved список areas

Этап 3. Геокодинг / геопривязка районов
Нужно определить, какой вид геоданных реально нужен `core_areas`:
- полигон,
- centroid,
- point,
- bbox,
- geojson,
- postgis geometry/geography,
- или другой формат.

После этого для каждого нового района нужно получить геоданные именно в том формате, который нужен текущей модели.

Если текущая модель районов хранит полигон:
- найти или собрать корректную геометрию
- сохранить её в нужном формате

Если текущая модель хранит точку / centroid:
- вычислить или получить корректный centroid района

Если модель хранит только название без геометрии:
- всё равно провести геопривязку и убедиться, что в проекте достаточно только текущих полей

Важно:
- не использовать “примерные” точки на глаз
- геоданные должны быть достаточны для текущих area-резолверов и фильтров карты

Результат этапа:
- набор готовых данных для вставки новых areas

Этап 4. Заведение новых городов и районов в БД
Нужно создать миграцию/сид/скрипт, который:
- добавляет новые записи в `core_cities`

- делает всё идемпотентно
- не создаёт дубли при повторном запуске
- не ломает существующие city/area ids Москвы и Санкт-Петербурга

При создании новых cities/areas нужно:
- использовать тот же стиль naming, slugging, ordering и прочих полей, что уже принят для Москвы и Санкт-Петербурга
- если в проекте уже есть conventions по id generation, slug, sort_order, display_name, normalized_name — использовать их

Этап 5. Актуализация серверной логики под новые города

Здесь нужен принцип:

Этап 6. Импорт новых мест
После того как cities и areas заведены, нужно загрузить новые места новых городов так же, как сейчас в системе заведены места Москвы и Санкт-Петербурга.

Использовать уже подготовленные входные geocoded CSV по городам.
Импорт мест должен идти по уже существующему канону проекта:
- не ломать `core_places`
- enrichment вести через `place_enrichment`
- фото для мест привязывать через `place_enrichment.photos`
- при отсутствии реального фото использовать category-placeholder
- мультикатегорийность вести через `core_place_category_links`
- если место уже существует и матчится по названию+адресу, не создавать дубль, а обновлять существующую запись и сливать типы

Логика импорта мест:
1. Для каждого места искать матч в существующей БД по нормализованным названию и адресу.
2. Если матч найден:
   - обновить более актуальными данными
   - обновить enrichment
   - обновить/добавить фото
   - слить типы заведения
3. Если матч не найден:
   - создать новое место
   - корректно назначить `area_id`
   - создать enrichment
   - создать category links
   - привязать фото или placeholder

Особо важно:
новые типы заведений уже есть/должны быть поддержаны:
- hookah
- karaoke
- bathhouse

Если в БД их ещё нет как category records — добавить корректно.
