# TZ2 — Нормализация данных: адреса, телефоны, сайты, фото

> Сессия 2 из 3
> Зависимости: TZ1 выполнена, staging_places_audit заполнена, нерелевантные места отфильтрованы

---

## Цель сессии

Для всех мест с verdict='keep' нормализовать данные из парсинга: адреса, телефоны, сайты. Отобрать лучшее фото для каждого места. Подготовить финальный staging для заливки в прод.

---

## Шаг 1: Создать staging-таблицу нормализованных данных

```sql
CREATE TABLE staging_places_normalized (
  place_id UUID PRIMARY KEY,
  -- Из парсинга (нормализованные)
  normalized_address TEXT,
  normalized_phone TEXT,        -- E.164 формат или NULL
  normalized_website TEXT,       -- валидный домен или NULL
  yandex_rating NUMERIC,
  -- Фото
  selected_photo_index INT,      -- какое фото выбрано (1, 2 или 3)
  selected_photo_path TEXT,      -- локальный путь к выбранному файлу
  photo_needs_placeholder BOOLEAN DEFAULT FALSE, -- нет хорошего фото → плейсхолдер
  -- Мета
  processing_status TEXT DEFAULT 'pending', -- pending | processed | error
  notes TEXT
);
```

## Шаг 2: Нормализация адресов

### 2.1 Правила очистки адресов
Берём `yandex_address` из `place_search_results_raw.raw` и применяем:

1. **Обрезать после корпуса/строения**: убрать всё после последнего "корп.", "стр.", "к." если дальше идёт "подъезд", "этаж", "кв.", "офис", "пом."
2. **Regex для обрезки**:
   ```
   (,?\s*(подъезд|под\.|этаж|эт\.|кв\.|квартира|офис|оф\.|помещение|пом\.|комната|ком\.|каб\.).*)$
   ```
3. **Убрать "Россия," в начале** если есть
4. **Убрать индекс** в начале (6 цифр)
5. **Оставить**: город, улица, дом, корпус/строение

### 2.2 Сверка с core_places
- Если yandex_address пустой или "no_results" → использовать текущий адрес из core_places
- Если оба есть → предпочитать яндексовский (более полный и актуальный)

## Шаг 3: Нормализация телефонов

### 3.1 Правила
1. Взять `raw.phone` или первый из `raw.phones_all`
2. Убрать все нецифровые символы кроме `+`
3. Привести к формату `+7XXXXXXXXXX` (11 цифр для России)
4. Если номер начинается с `8` → заменить на `+7`
5. Если номер короче 11 цифр (без `+7`) → невалидный → NULL
6. Если номер длиннее 12 символов → невалидный → NULL

### 3.2 Валидация
- Номер должен быть `+7` + 10 цифр
- Всё остальное → NULL (лучше без телефона чем с кривым)

---

## Шаг 4: Нормализация сайтов

### 4.1 Правила
1. Взять `raw.website`
2. Убрать протокол, trailing slash, www.

### 4.2 Стоп-лист доменов (убираем → NULL)
- `2gis.ru`, `go.2gis.com`
- `yandex.ru`, `maps.yandex.ru`
- `vk.com`, `vk.me`
- `t.me`, `telegram.me`, `telegram.org`
- `instagram.com`, `facebook.com`
- `wa.me`, `whatsapp.com`
- `youtube.com`
- `tripadvisor.com`
- `booking.com`
- `zoon.ru`
- `flamp.ru`
- `restoclub.ru` (агрегатор, не сайт места)

### 4.3 Валидация
- Домен должен хоть как-то ассоциироваться с названием места (эвристика: хотя бы первые 3 символа или транслит)
- Если совсем не соответствует — оставляем, но помечаем для ревью
- Пустой или стоп-лист → NULL

## Шаг 5: Отбор фото

### 5.1 Анализ фото из парсинга
Для каждого места в scraped_photos/ есть от 0 до 3 фото:
- `{place_id}.webp` — первое фото
- `{place_id}_2.webp` — второе
- `{place_id}_3.webp` — третье

### 5.2 Критерии выбора лучшего фото
Задача: из 1-3 фото выбрать одно оптимальное. Приоритеты:

1. **Интерьер заведения** — лучший вариант (зал, барная стойка, столики)
2. **Фасад** — приемлемо если интерьера нет
3. **Еда/напитки** — приемлемо для ресторанов/кафе
4. **НЕ использовать**: фото с крупными лицами людей (где можно различить черты)

### 5.3 Автоматический отбор
- Если 1 фото → оно и есть выбор
- Если 2-3 фото → нужна эвристика или ручной проход
- **Вариант**: использовать размер файла + aspect ratio как proxy (интерьерные фото обычно больше по весу)
- **Вариант**: первое фото в Яндексе обычно самое релевантное → по умолчанию выбираем первое

### 5.4 Постобработка выбранного фото
- Лёгкая коррекция: +5-10% резкости, +5-10% насыщенности (через sharp)
- НЕ должно визуально портить — минимальное отличие от оригинала
- Формат: webp, quality 82 (уже так в парсинге)
- Финальное имя файла для Storage: `scraped_photos/{place_id}.webp`

### 5.5 Места без фото
- Если в парсинге нет фото → photo_needs_placeholder=TRUE
- На проде будет использоваться плейсхолдер по типу из category_placeholder.dart

## Шаг 6: Скрипт обработки

### 6.1 Создать Node.js/Python скрипт
`scripts/normalize_places.js` (или .py):

1. Читает все записи из place_search_results_raw где place_id в staging_places_audit с verdict='keep'
2. Для каждого места:
   - Нормализует адрес
   - Нормализует телефон
   - Нормализует сайт
   - Определяет лучшее фото
   - Применяет постобработку к выбранному фото
3. Записывает результат в staging_places_normalized

### 6.2 Отчёт после обработки
- Сколько мест с валидным телефоном / без
- Сколько мест с валидным сайтом / без
- Сколько мест с фото / с плейсхолдером
- Сколько адресов было обрезано (подъезды/этажи)

## Выходные артефакты

1. **staging_places_normalized** — таблица с нормализованными данными на каждое место
