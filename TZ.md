ТЗ: система бонусных токенов Centry
1. Контекст проекта
Centry — мобильное приложение на Flutter с backend на Supabase / Postgres, построенное вокруг социальных сценариев, связанных с местами, друзьями, планами и пользовательской активностью.
Ключевой архитектурный принцип проекта: server-first. Это означает:
* сервер является единственным источником истины
* все продуктовые факты, статусы, права, агрегаты и итоговые состояния считаются на сервере
* клиент не должен самостоятельно вычислять бизнес-логику
* клиент получает от сервера уже готовые snapshot-данные и только рендерит их
Важная доменная основа проекта: все продуктовые сценарии строятся вокруг app_user_id, а не вокруг сырого auth.uid().
Система бонусных токенов должна быть встроена в эту архитектуру как полноценный серверный доменный блок, а не как локальная клиентская надстройка.

2. Цель задачи
Нужно спроектировать и реализовать полноценную систему бонусных токенов в Centry.
На текущий момент:
* серверной бонусной системы не существует
* баланса токенов не существует
* счетов / ledger / истории начислений не существует
* на клиенте есть только отрисованное окно, не привязанное к реальным серверным данным
Нужно сделать:
1. продуктовую модель начисления токенов
2. серверную модель хранения и учёта
3. механику начисления
4. защиту от дублей и повторных начислений
5. баланс токенов
6. историю начислений
7. серверные snapshot / RPC для клиента
8. клиентскую интеграцию
9. подготовить основу под будущие расширения системы

3. Продуктовая цель системы
Бонусные токены нужны для поощрения пользователей за реальный вклад в развитие Centry.
Начисление должно происходить не за нажатия в UI, а только за подтверждённые сервером продуктовые события.
Система должна мотивировать пользователей:
* создавать и доводить планы до завершения
* участвовать в голосовании
* приглашать новых пользователей через инвайты в планы
* добавлять качественные новые места
* обогащать базу мест Centry

4. Канон системы начисления
4.1. Основной принцип
Токены начисляются только за серверно подтверждённые доменные события.
Запрещено строить начисления на:
* клиентских действиях
* локальных флагах клиента
* UI-фактах
* предположениях клиента
* событиях, не доведённых до доменного результата

4.2. Набор механик V1
В первой версии системы должны быть реализованы 6 механик начисления.
1. Успешно завершённый план — 10 токенов
Получатель: создатель плана
Начисляется, если план:
* прошёл полный сценарий
* имеет итоговую дату и итоговое место
* был успешно завершён сервером

2. Голосование за дату в завершённом плане — 1 токен
Получатель: пользователь, голосовавший за дату
Начисляется, если:
* пользователь действительно проголосовал за дату
* план в итоге был успешно завершён
Ограничение:
* максимум 1 начисление за голосование по дате на пользователя в рамках одного плана

3. Голосование за место в завершённом плане — 1 токен
Получатель: пользователь, голосовавший за место
Начисляется, если:
* пользователь действительно проголосовал за место
* план в итоге был успешно завершён
Ограничение:
* максимум 1 начисление за голосование по месту на пользователя в рамках одного плана

4. Регистрация пользователя по приглашению в план — 15 токенов
Получатель: пользователь, который создал план и отправил приглашение
Начисляется, если:
* существовал инвайт в план
* новый пользователь зарегистрировался в Centry по этому инвайту
* регистрация успешно завершена
* связь регистрации с инвайтом канонично подтверждена сервером
Важно:
* у проекта нет общей реферальной системы
* речь только о регистрации пользователя, приглашённого по инвайту в план

5. Добавленное место прошло модерацию — 12 токенов
Получатель: автор добавленного места
Начисляется, если:
* пользователь добавил новое место
* место прошло модерацию
* место стало валидным объектом системы
Ограничение:
* одно начисление на одно место

6. Место стало финальным в завершённом плане — 5 токенов
Получатель: автор места
Начисляется, если:
* место было добавлено пользователем
* это место стало итоговым местом завершённого плана
Важно:
* начисление происходит каждый раз, когда это место становится финальным местом нового успешно завершённого плана

4.3. Будущие механики V1.1 — пока не реализовывать
Нужно сразу заложить архитектурную расширяемость, но не реализовывать в этом ТЗ:
* первый успешно завершённый план
* первое место, прошедшее модерацию
* первый приглашённый пользователь

5. Общие продуктовые правила
1. Токены начисляются только автоматически сервером.
2. Клиент не инициирует начисление напрямую.
3. Каждое начисление должно быть:
    * трассируемым
    * объяснимым
    * идемпотентным
4. Пользователь должен видеть:
    * текущий баланс
    * историю начислений
    * понятное описание, за что начислены токены
5. Начисления должны быть защищены от дублей и повторного исполнения.
6. Система должна быть готова к будущему появлению:
    * списаний
    * ручных корректировок
    * акций / кампаний
    * достижений / одноразовых бонусов

6. Обязательный этап: исследование перед реализацией
До внесения изменений архитектор/разработчик обязан провести исследование текущей серверной и клиентской базы.
6.1. Серверное исследование
Нужно изучить и зафиксировать:
Пользователи
* как устроена доменная модель app_user_id
* как связаны auth user и app user
* какие таблицы являются каноническими для пользователя
Планы
* где хранится план
* где хранится создатель плана
* как определяется финальное состояние плана
* где хранятся итоговые дата и место
* какая функция / набор функций выполняет финализацию плана
* есть ли уже канонический признак "успешно завершён"
Голосования
* где хранятся голоса за даты
* где хранятся голоса за места
* есть ли soft delete / пересчёты / перевыбор
* какие структуры считать каноническими для факта голосования
Инвайты в план
* где хранится инвайт
* кто является владельцем инвайта
* как определяется, кто пригласил пользователя
* как фиксируется вход нового пользователя через инвайт
* как сейчас связывается регистрация и плановый invite-flow
Места
* где хранится место
* где хранится автор пользовательского места
* как устроен moderation flow
* какой статус или событие означает "место прошло модерацию"
* как отличить пользовательское место от системного / импортированного / уже существовавшего
Потенциальные точки встраивания начислений
Нужно определить:
* в какие существующие RPC / функции безопаснее встроить начисление
* где лучше делать post-action начисления
* где лучше использовать server-side helper functions
* какие функции уже являются каноническими точками завершения бизнес-сценария

6.2. Клиентское исследование
Нужно изучить:
* какой экран сейчас отрисовывает бонусную систему
* какие клиентские модели уже есть
* какие DTO / repository / state management блоки нужны для интеграции
* есть ли уже пользовательский профиль / хедер / экран кошелька / экран бонусов
* где должен отображаться баланс
* где должна отображаться история начислений
* нужно ли обновление в realtime или достаточно pull / refresh

6.3. Результат исследования
До начала кодинга должен быть сформирован короткий технический вывод:
1. какие текущие таблицы и функции используются как канонические источники событий
2. какие таблицы и функции нужно доработать
3. какие новые таблицы, RPC и snapshot-методы нужно создать
4. в каких точках будут происходить начисления
5. как обеспечивается идемпотентность

7. Техническая модель: как должна быть устроена серверная система
7.1. Нельзя делать только одно поле balance
Запрещено реализовывать систему как:
* одно число balance в профиле
* прибавление/вычитание без истории
* клиентское суммирование из произвольных источников
Нужна ledger-модель.

7.2. Обязательные сущности
Нужно реализовать как минимум следующие сущности.
A. Счёт пользователя / бонусный аккаунт
Отдельная сущность бонусного счёта пользователя.
Назначение:
* хранить текущий агрегированный баланс
* быть привязанной к app_user_id
* служить точкой входа для чтения баланса
Возможные поля:
* id
* app_user_id
* current_balance
* created_at
* updated_at
Важно:
* current_balance — это кэш/агрегат
* источником истины для аудита остаётся ledger

B. Журнал операций / ledger
Главная таблица системы.
Каждая строка — отдельная бонусная операция.
Пример состава полей:
* id
* app_user_id
* account_id
* entry_type (credit, в будущем debit, adjustment)
* amount
* reward_code
* source_entity_type
* source_entity_id
* source_context_json
* idempotency_key
* description
* created_at
* created_by_system
* status при необходимости
Это должен быть канонический журнал начислений.

C. Таблица reward-событий / reward rules catalog
Нужна таблица или enum-слой для типов начислений.
В V1 должны поддерживаться коды:
* PLAN_COMPLETED_OWNER
* PLAN_VOTE_DATE_COMPLETED
* PLAN_VOTE_PLACE_COMPLETED
* PLAN_INVITE_REGISTRATION
* PLACE_APPROVED_MODERATION
* PLACE_BECAME_FINAL
Задача:
* стандартизировать reward-коды
* использовать их в ledger
* использовать их в клиентских текстах
* подготовить основу под конфигурируемость

D. Таблица или слой конфигурации reward amounts
Размеры наград не должны быть размазаны по коду.
Нужен канонический серверный слой конфигурации:
* reward code
* amount
* is_enabled
* effective_from / effective_to при желании
* metadata
Для V1 значения:
* завершённый план — 10
* голос дата — 1
* голос место — 1
* регистрация по инвайту — 15
* место прошло модерацию — 12
* место стало финальным — 5
Если таблица конфигурации кажется избыточной, допустим минимальный вариант: константный серверный слой в SQL/функции. Но лучше сразу делать через конфигурационную таблицу.

7.3. Идемпотентность — обязательна
Каждое начисление должно быть защищено от повторного выполнения.
Для этого у каждой операции начисления должен быть устойчивый idempotency key.
Примеры:
* завершённый план для создателя PLAN_COMPLETED_OWNER:<plan_id>:<owner_app_user_id>
* голос за дату PLAN_VOTE_DATE_COMPLETED:<plan_id>:<app_user_id>
* голос за место PLAN_VOTE_PLACE_COMPLETED:<plan_id>:<app_user_id>
* регистрация по инвайту PLAN_INVITE_REGISTRATION:<invite_id>:<inviter_app_user_id>:<new_app_user_id>
* место прошло модерацию PLACE_APPROVED_MODERATION:<place_id>:<creator_app_user_id>
* место стало финальным PLACE_BECAME_FINAL:<plan_id>:<place_id>:<creator_app_user_id>
На уровне БД должен быть unique constraint по idempotency_key.

7.4. Баланс
Баланс пользователя должен существовать на сервере и считаться канонически.
Возможны два подхода:
Подход 1 — агрегированное поле + ledger
* ledger хранит все операции
* current_balance обновляется при каждой вставке
Это предпочтительный вариант для клиента.
Подход 2 — расчёт по сумме ledger
* баланс суммируется из операций на лету
Как единственный механизм — хуже. Можно использовать как fallback/верификацию, но не как основную модель для production UI.

8. Начисления: где и как именно должны происходить
8.1. Начисления запрещено делать с клиента
Клиент не должен:
* вызывать add_tokens
* передавать сумму начисления
* решать, что событие произошло
* проверять eligibility
Все начисления — строго на сервере.

8.2. Начисления должны происходить в канонических точках доменных сценариев
A. Завершённый план
Начисление должно происходить там, где сервер канонически фиксирует завершение плана.
В этой точке нужно:
1. определить создателя плана
2. начислить 10 токенов создателю
3. найти пользователей, голосовавших за дату
4. начислить каждому eligible пользователю 1 токен
5. найти пользователей, голосовавших за место
6. начислить каждому eligible пользователю 1 токен
7. если итоговое место пользовательское — начислить автору места 5 токенов
Это либо:
* делается внутри канонической finalize-function
* либо вызывается helper-функция после успешной финализации
Важно:
* начисление должно происходить только после успешной фиксации финального состояния
* повторный вызов не должен дублировать начисления

B. Регистрация по инвайту в план
Начисление должно происходить там, где сервер канонически завершает onboarding пользователя, пришедшего по плановому invite-flow.
Нужно:
1. определить факт регистрации нового пользователя
2. определить, что регистрация атрибутирована к инвайту в план
3. определить владельца инвайта / план / создателя плана
4. начислить 15 токенов пригласившему
Важно:
* начисление только после успешного создания доменного пользователя
* только при валидной связке с invite
* без дублей

C. Место прошло модерацию
Начисление должно происходить в точке, где сервер подтверждает принятие пользовательского места модерацией.
Нужно:
1. определить автора места
2. убедиться, что это именно пользовательское место
3. убедиться, что оно прошло модерацию
4. начислить 12 токенов
Важно:
* при повторной модерации / повторном апдейте не дублировать
* если место отклонено, начисления нет

9. Обязательные server-side helper functions
Нужно спроектировать отдельный слой серверных helper-функций для бонусной системы.
Минимально нужны функции уровня:
9.1. ensure_bonus_account(app_user_id)
Создаёт бонусный счёт при отсутствии.
9.2. grant_bonus_if_absent(...)
Универсальная функция безопасного начисления:
* принимает app_user_id
* reward_code
* amount
* source_entity_type
* source_entity_id
* idempotency_key
* description
* context_json
Функция:
* проверяет/создаёт аккаунт
* вставляет запись в ledger
* обновляет баланс
* является идемпотентной
9.3. process_plan_completion_rewards(plan_id)
Вызывается из канонической финализации плана и выполняет все начисления по плану.
9.4. process_invite_registration_reward(invite_id, new_app_user_id)
Начисление за регистрацию по инвайту.
9.5. process_place_approved_reward(place_id)
Начисление за прошедшее модерацию место.

10. RLS и безопасность
Нужно сохранить канон безопасности проекта.
Требования:
* пользователь может читать только свой бонусный аккаунт и свою историю начислений
* обычный клиент не может создавать, менять или удалять bonus ledger entries
* начисления выполняются только сервером / security definer функциями / trusted backend flow
* клиент не может подделать reward_code, amount, source ids
Нужно продумать:
* RLS для бонусного аккаунта
* RLS для ledger
* какие RPC должны быть security definer
* как в них прокидывается и валидируется app.current_app_user_id

11. API / RPC / snapshot для клиента
Клиенту нельзя отдавать сырые серверные куски в случайных местах. Нужен понятный серверный контракт.
Нужно реализовать минимум:
11.1. RPC / view для получения бонусного summary
Должен возвращать:
* current_balance
* возможно total_earned
* возможно recent_transactions_count
* при необходимости краткие counters
Минимум — баланс.

11.2. RPC / view для списка операций
Должен возвращать paginated history:
* id
* amount
* reward_code
* title
* subtitle / description
* created_at
* source_entity_type
* source_entity_id
Тексты должны быть серверно нормализованы либо сервер должен отдавать достаточно данных, чтобы клиент тупо рендерил по каноничному mapping.
Лучше:
* сервер отдаёт нормализованный тип + description
* клиент рендерит по таблице безопасных UI-моделей

11.3. RPC / view для правил начисления
Опционально, но желательно.
Клиенту можно отдавать список текущих reward rules:
* code
* amount
* display_title
* display_description
* sort_order
* is_enabled
Это позволит не хардкодить правила в клиенте навсегда.
Если не делать в V1, допустим временный статический клиентский экран правил. Но архитектурно лучше сразу иметь серверный источник.

12. Клиентская часть: что нужно реализовать
Клиент остаётся тупым UI и работает только с серверными данными.
12.1. Экран бонусов
Нужно довести уже существующее отрисованное окно до рабочего состояния.
Экран должен показывать:
* текущий баланс токенов
* список последних начислений
* правила начисления бонусов
* состояние загрузки
* пустое состояние
* состояние ошибки

12.2. Источник данных клиента
Нужно реализовать:
* repository слой
* DTO модели под summary и ledger history
* маппинг DTO → UI model
* экран / state management
* pull-to-refresh или другой refresh-механизм

12.3. Что клиент не должен делать
Клиент не должен:
* считать баланс сам по нескольким экранам
* вычислять eligibility
* определять факт начисления
* запускать начисление кнопкой
* сам решать, было ли место финальным или план завершённым

12.4. Где должен отображаться баланс
Архитектор должен исследовать текущий UX и определить:
* нужен ли баланс только на экране бонусов
* нужен ли баланс в профиле / хедере / другом блоке
* нужна ли мини-карточка с балансом в других местах
В рамках V1 минимум:
* рабочий экран бонусов с актуальным балансом

12.5. Реакция клиента на новые начисления
В первой версии допустим любой из вариантов:
* обновление по pull/refresh
* обновление после входа на экран
* soft refresh при resume
* realtime позже
Realtime для бонусной системы в V1 не обязателен, если он усложняет реализацию.

13. UX-требования к истории операций
Каждая операция должна быть понятна пользователю.
История должна показывать человеческий смысл начисления.
Примеры отображения:
* «Успешно завершённый план» — +10
* «Голосование за дату в завершённом плане» — +1
* «Голосование за место в завершённом плане» — +1
* «Регистрация пользователя по приглашению в план» — +15
* «Место прошло модерацию» — +12
* «Ваше место стало финальным местом встречи» — +5
Если возможно, добавить вторую строку:
* название плана
* название места
* дата начисления
Но это необязательная надстройка. Главное — чтобы базовая история уже была рабочей и понятной.

14. Антидубли и антиабьюз — обязательный минимум
Для V1 обязательно заложить как минимум следующее.
14.1. Идемпотентность
Все начисления должны быть защищены уникальными idempotency keys.
14.2. Начисление только после серверного подтверждения
Никаких pre-reward до завершения сценария.
14.3. Ограничения по голосованию
Только одно начисление:
* за дату на пользователя на план
* за место на пользователя на план
14.4. За место после модерации
Только одно начисление на место.
14.5. За финальное место — на каждый завершённый план
Но только если этот план действительно новый уникальный завершённый план.

15. Что может потребовать доработки существующих таблиц и функций
Так как блок реализуется с нуля, допускается и ожидается модернизация существующей серверной схемы.
Архитектор должен проверить, нужны ли изменения в:
* таблицах планов
* таблицах голосований
* таблицах мест
* таблицах place submissions / moderation
* таблицах invite-flow / registration attribution
* RPC финализации планов
* RPC/функциях модерации места
* функциях onboarding / invite acceptance / registration completion
Нужно определить:
1. достаточно ли текущих полей для канонической атрибуции reward-событий
2. есть ли все нужные foreign keys / references
3. нет ли дыр, где невозможно доказуемо определить получателя бонуса
4. нужно ли добавить новые поля, индексы или служебные таблицы

16. Требования к миграциям
Нужно подготовить серверную часть как нормальный production-grade блок.
Миграции должны:
* создавать новые таблицы бонусной системы
* создавать индексы
* создавать constraints
* создавать unique constraints на idempotency
* создавать RLS policies
* создавать функции начисления
* при необходимости модифицировать существующие функции и таблицы
* быть безопасными для повторяемого применения в окружениях

17. Требования к тестированию и проверке
После реализации нужно проверить все 6 сценариев начисления.
Обязательные тест-кейсы
1. Завершение плана
* план завершён
* создатель получил 10
* голосовавшие за дату получили по 1
* голосовавшие за место получили по 1
* если итоговое место пользовательское — автор получил 5
2. Повторная финализация / повторный вызов
* повторный вызов не дублирует начисления
3. Регистрация по инвайту
* новый пользователь зарегистрирован по invite-flow
* пригласивший получил 15
* повторный обработчик не создал дубль
4. Модерация места
* пользовательское место одобрено
* автор получил 12
* повторная обработка не даёт дубль
5. Финальное место
* одно и то же место стало финальным в двух разных завершённых планах
* автор получил по 5 за каждый из них
6. RLS
* пользователь видит только свои операции и свой баланс
* клиент не может сам записывать ledger

18. Нефункциональные требования
1. Реализация должна быть расширяемой.
2. Система должна поддерживать будущие reward types без переписывания основы.
3. Система должна быть пригодна для аудита.
4. История операций не должна теряться.
5. Решение не должно ломать существующие сценарии планов, мест и invite-flow.
6. Все изменения должны встраиваться минимально и безопасно в текущую архитектуру.

19. Что должно быть результатом работы
Архитектор/разработчик должен выдать завершённый результат, включающий:
Сервер
* новые таблицы бонусной системы
* миграции
* индексы и constraints
* RLS
* helper functions
* встраивание начислений в канонические доменные точки
* RPC / snapshot для клиента
* рабочий баланс
* рабочий ledger
* защиту от дублей
Клиент
* рабочий экран бонусов
* получение реального баланса с сервера
* получение истории начислений с сервера
* отображение правил начисления
* корректные loading / empty / error states
Документация / фиксация
* список reward codes
* правила начисления
* описание точек начисления

* описание того, какие существующие функции были доработаны

21. Отдельно зафиксировать для архитектора
Это критично:
* не делать клиент источником истины
* не делать баланс без ledger
* не начислять токены из UI
* не размазывать логику по Flutter
* не привязывать продуктовую механику к auth.uid вместо app_user_id
* не делать награды без идемпотентности
* не ломать существующие канонические серверные сценарии

22. Краткая итоговая формулировка задачи
Нужно с нуля реализовать в Centry полноценную server-first систему бонусных токенов с балансом, историей начислений, серверным ledger-учётом, защитой от дублей, интеграцией в доменные сценарии планов / инвайтов / модерации мест и с рабочим клиентским экраном, который получает реальные данные с сервера и не содержит бизнес-логики начисления.
