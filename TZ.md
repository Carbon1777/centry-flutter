ТЗ: блоки профиля пользователя для приложения офлайн-отдыха и совместных встреч
1. Общая идея
Профиль пользователя в приложении должен помогать другим людям быстро понять:
* как человек любит отдыхать,
* насколько совпадает формат досуга,
* насколько комфортно и безопасно планировать с ним совместный отдых.
Профиль не должен быть перегружен длинными текстами. Основной акцент делаем на структурированных блоках с иконками и короткими подписями. Свободное описание пользователя допускается как опциональный небольшой блок, но не является основным источником информации.
Фото и видео в это ТЗ не входят, так как для них уже определено отдельное место.

2. Состав блоков профиля
Ниже перечислены блоки, которые должны быть реализованы в профиле.

* если ничего не выбрано, блок можно не показывать либо показывать в пустом состоянии только владельцу профиля.

4. UX-требования
4.1. Общие требования к отображению
* Все блоки должны выглядеть легко и компактно.
* Основной формат отображения — чипсы / карточки / строки с иконкой + коротким текстом.
* Не использовать длинные абзацы.
* Профиль должен быстро сканироваться глазами за несколько секунд.
4.2. Требования к редактированию
* Все структурированные блоки редактируются через экран редактирования профиля.
* Для множественного выбора нужно явно показывать лимит:
    * “Можно выбрать до 3”
    * “Можно выбрать до 2”
* После достижения лимита новые варианты не выбираются, либо предлагается сначала снять один из выбранных.
4.3. Пустые состояния
* Если владелец профиля ничего не заполнил — показывать мягкие подсказки на редактировании.
* Для чужого профиля пустые блоки лучше не показывать вообще.
* Не должно быть ощущения “битого” или “пустого” экрана.

5. Требования к данным и архитектуре
5.1. Архитектурный принцип
Хранить не свободные тексты-лейблы, а канонические ключи значений. Иконки и тексты должны быть привязаны к этим ключам.
Пример:
* rest_format_dining
* rest_format_bars
* rest_dislike_noise
* social_small_group
* tempo_spontaneous
То есть сервер/БД хранит enum-like keys, а клиент рендерит их через локальный словарь или серверный справочник.
5.2. Рекомендуемая модель
У пользователя должны храниться поля примерно такого типа:
* rest_preferences[] — массив до 3 значений
* rest_dislikes[] — массив до 3 значений
* social_format — одно значение
* rest_tempo — одно значение
* meeting_time_preferences[] — массив до 2 значений
* vibe — одно значение
* short_bio — короткая строка
5.3. Валидация
На сервере обязательно валидировать:
* допустимые ключи;
* лимиты выбора;
* длину текстового поля;
* отсутствие дублей в массивах.

6. Требования к текстам и иконкам
6.1. Иконки
* Иконки должны быть визуально нейтральные, дружелюбные, без излишней “детскости”.
* Иконка — вспомогательная, а не основная сущность.
* Иконки должны быть единообразны по стилю во всех блоках.
6.2. Тексты
* Все тексты короткие, понятные, разговорные, без канцелярита.
* Не использовать тяжёлые формулировки типа “предпочитаю социальное взаимодействие в узком кругу”.
* Формулировки должны быть живыми и человеческими.

7. Что не делать
* Не делать профиль в формате классической соцсети с большим “О себе”.
* Не превращать блоки в психологический тест.
* Не делать слишком много текста в одном блоке.
* Не использовать негативные или токсичные формулировки.
* Не хранить выбранные значения только как display text — только как ключи.

8. Ожидаемый итог
После реализации профиль пользователя должен быстро отвечать на вопросы:
* какой формат отдыха любит человек;
* чего он избегает;
* в каком составе ему комфортнее;
* насколько совпадает стиль отдыха;
* есть ли ощущение “мой человек / не мой человек” без необходимости читать длинное описание.

Обязательные к реализации блоки:

3. Комфортный формат компании — 1 вариант
4. Темп отдыха — 1 вариант

6. Мой вайб — 1 вариант

 ТЗ: компоновка блоков профиля пользователя
1. Общий принцип
Профиль строится по модели:
* вертикальный скролл
* блоки идут сверху вниз
* без сложных сеток
* без фиксированных “пустых мест”
Главное правило:
UI должен быть контент-зависимым (content-driven), а не сеточно-зависимым

2. Структура экрана профиля
Порядок блоков:
1. (вне ТЗ) Фото / видео
2. Как люблю отдыхать
3. Что не люблю
4. Компактный блок характеристик
    * Формат компании
    * Темп отдыха
    * Время встреч
    * Мой вайб
5. Коротко о себе (если заполнено)

3. Контейнер профиля
Общие отступы
* горизонтальные: 16px
* вертикальные между блоками: 16–20px
Ширина
* 100% экрана
* внутренние блоки не должны прилипать к краям

4. Блоки с множественным выбором (КЛЮЧЕВОЕ)
Блоки:
* Как люблю отдыхать
* Что не люблю
* Когда удобнее встречаться

4.1. Заголовок блока
Формат:
* текст заголовка
* размер: medium / semibold
* отступ снизу: 8px

4.2. Контейнер значений
Тип layout:
👉 Wrap / Flow layout (аналог flex-wrap)

4.3. Чип (элемент)
Каждый выбранный пункт отображается как чип:
Состав:
* иконка
* текст (1 строка)
Размеры:
* высота: 28–32px
* padding: 8–12px
* border-radius: 16px+
Поведение:
* ширина по контенту, НЕ фиксированная
* текст не переносится
* если не влезает — перенос на следующую строку

4.4. Расположение
* элементы идут слева направо
* между элементами:
    * горизонтальный gap: 8px
    * вертикальный gap: 8px
* автоматический перенос строк

4.5. Критично
❌ НЕЛЬЗЯ:
* фиксировать 3 колонки
* делать placeholder под “пустые” элементы
* растягивать чипы на всю ширину
* выравнивать сеткой
✔ ДОЛЖНО БЫТЬ:
* 1 элемент → одна аккуратная плашка
* 2 элемента → 2 плашки рядом
* 3 элемента → перенос при необходимости

5. Компактный блок характеристик
Блоки:
* Формат компании
* Темп отдыха
* Когда встречается
* Мой вайб

5.1. Формат
Это НЕ чипсы, а строки характеристик

5.2. Контейнер
Тип:
* вертикальный список
Отступы:
* между строками: 10–12px

5.3. Элемент строки
Каждая строка:

[иконка]  Название: Значение


Пример:
* 👥 Формат: Небольшая компания
* ⚡ Темп: Гибкий
* 🌇 Время: Вечер, Поздний вечер
* 😎 Вайб: Легкий на подъем

5.4. Поведение
* всегда 1 строка (если помещается)
* допускается перенос строки для длинного значения
* иконка фиксированного размера
* текст — гибкий

5.5. Альтернатива (если нужен “дороже” UI)
Можно сделать не строки, а плашки full-width:
* одна характеристика = одна карточка
* но это увеличивает высоту экрана
Рекомендуется базовый вариант (строки).

6. Блок "Коротко о себе"
Условия отображения
* показывается только если заполнен

Формат
* заголовок
* текст до 2–3 строк
* без expand/collapse на MVP

Отступы
* от заголовка: 8px

7. Логика отображения блоков
7.1. Пустые блоки
❌ Не показывать блок, если:
* нет выбранных значений
* нет текста
✔ Исключение:
* на экране редактирования

7.2. Приоритет
Сначала показываются наиболее важные блоки:
1. Как люблю отдыхать
2. Что не люблю

8. Поведение при разных количествах элементов
1 элемент
* занимает одну строку
* выравнивание по левому краю
2 элемента
* в одну строку, если помещаются
* иначе перенос
3 элемента
* авто перенос
* никаких “дыр”

9. Важные UX-принципы
1. Нет визуальных дыр
2. Нет фиксированных колонок
3. Контент управляет размером
4. Минимум визуального шума
5. Быстрое сканирование (2–3 секунды)

11. Критические ошибки, которые нельзя допустить
* сетка 3x3 с пустыми элементами
* одинаковые карточки фиксированной ширины
* слишком большие блоки
* перегруженность текстом
* отсутствие адаптивности

12. Резюме для архитектора
Использовать:
* wrap layout для множественных значений
* list layout для одиночных
* content-driven sizing
* server-driven значения (через ключи)
