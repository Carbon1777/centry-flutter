ТЗ: удаление аккаунта в Centry
1. Контекст
Centry — Flutter + Supabase / Postgres, архитектура server-first. Источник истины — сервер. Клиент не принимает решений о составе удаляемых данных, правах, статусах и сценариях очистки. Все правила удаления аккаунта, обезличивания, блокировки и очистки должны жить на сервере.
В приложении уже есть/планируются:
* профиль пользователя
* планы, участники, голосования по датам и местам
* социальные связи
* мессенджер
* пользовательские места
* фото/видео контент
* бонусная система
* в будущем подписки и пополнение баланса
Поэтому удаление аккаунта должно быть реализовано как отдельный доменный серверный сценарий, а не как локальная кнопка “delete row from users”.

2. Цель задачи
Реализовать полноценный сценарий удаления аккаунта, при котором:
* пользователь может инициировать удаление аккаунта из клиента
* аккаунт немедленно перестаёт участвовать в продуктовых сценариях
* персональные данные удаляются или обезличиваются по серверным правилам
* система не ломает целостность планов, сообщений, мест, бонусного ledger и будущих платёжных сущностей
* все действия логируются и могут быть подтверждены
* решение расширяемо и не требует переписывать половину базы при появлении новых модулей

3. Рекомендуемый канон удаления
3.1. Общий подход
Реализовать двухфазное удаление:
Фаза A. Немедленная деактивация
Сразу после подтверждения пользователем:
* аккаунт переводится в состояние PENDING_DELETION или аналогичное
* пользователь разлогинивается
* профиль скрывается из продукта
* пользователь перестаёт:
    * входить в приложение
    * отображаться в поиске
    * участвовать в приглашениях
    * быть доступным как активный собеседник/участник
    * инициировать новые действия
Фаза B. Фоновая серверная обработка удаления
Отдельный серверный pipeline:
* удаляет PII
* обезличивает нужные продуктовые следы
* чистит связи
* обновляет derived/snapshot данные
* ставит итоговый статус DELETED
Такой подход лучше, чем “в один SQL-запрос всё снести”.

3.2. Что выбрать: полное удаление или обезличивание
Решение для Centry:
не делать только один из вариантов; делать гибрид.
Жёстко удалять:
* телефон
* email
* имя/никнейм, если он является персональными данными текущего аккаунта
* фото профиля
* приватные настройки
* push tokens / device tokens
* active sessions / refresh tokens
* приватные метаданные профиля
* черновики, если они персональные
* приватные media-файлы пользователя, если они не нужны для целостности чужого контента
Обезличивать:
* сообщения в чатах
* авторство в старых событиях/лентах, если полное удаление сломает историю
* участие в старых завершённых планах
* записи в голосованиях, если они нужны для консистентности завершённых планов/аналитики
* ledger бонусов, если он нужен для финансовой/аудиторской непротиворечивости
* историю moderation и abuse/security logs
Оставлять по retention-policy:
* минимальные системные журналы безопасности
* юридически значимые записи по платежам / подпискам / возвратам
* записи о согласии с документами
* антифрод/безопасность записи
* операции внутреннего баланса, если они будут связаны с покупками и спорами

4. Почему не делать полное каскадное физическое удаление всего
Для Centry это плохой вариант, потому что он:
* сломает завершённые планы и историю голосований
* сломает чаты у других пользователей
* создаст дыры в ledger бонусов и будущих платежах
* потребует каждый раз переписывать всё при появлении нового блока
* повысит риск потери юридически значимых данных
Поэтому архитектурно правильнее строить центральный orchestration-flow удаления, где каждый модуль знает, как себя “очистить” или “обезличить”.

5. Продуктовое поведение
5.1. Где находится кнопка
Кнопка «Удалить аккаунт» находится:
* Профиль
* Настройки
* либо блок “Приватность и данные”
Финальное размещение определить в UX, но с точки зрения продукта для пользователя это должен быть явно доступный, но не случайный сценарий.

5.2. UX flow
Экран 1. Предупреждение
Показать экран с объяснением:
Что произойдёт:
* профиль будет удалён
* доступ к аккаунту будет закрыт
* личные данные будут удалены или обезличены
* часть информации может остаться в обезличенном виде в завершённых планах, сообщениях, бонусной и платёжной истории, если это необходимо для целостности сервиса и соблюдения обязанностей сервиса
Экран 2. Подтверждение
Подтверждение через:
* чекбокс “Я понимаю последствия удаления аккаунта”
* кнопка “Удалить аккаунт”
Желательно:
* повторный ввод кода/пароля/подтверждения, если есть чувствительная аутентификация
* либо системный re-auth flow, если это поддерживается auth-моделью

5.3. Поведение после подтверждения
После подтверждения:
* показываем “Аккаунт удаляется”
* выполняем серверный RPC
* по успеху:
    * разлогин
    * очистка локальных данных
    * переход на стартовый экран / экран онбординга

6. Серверный канон
6.1. Удаление аккаунта не должно делаться напрямую из клиента
Запрещено:
* удалять пользователя клиентом через raw table write
* выполнять набор случайных SQL-операций с клиента
* решать на клиенте, что удалять, а что обезличивать
Только серверная каноническая функция.

6.2. Главная серверная функция
Нужна каноническая RPC / server function уровня:
request_account_deletion_v1(p_app_user_id uuid, p_reason text default null)
Эта функция:
1. валидирует пользователя
2. проверяет право удалить именно свой аккаунт
3. проверяет ограничения
4. переводит аккаунт в статус pending deletion
5. создаёт deletion job
6. инвалидирует доступ/сессии
7. пишет audit log
8. возвращает клиенту результат

6.3. Фоновая функция/воркер
Нужен отдельный обработчик:
process_account_deletion_job_v1(p_job_id uuid)
Он:
* проходит по всем доменным блокам
* удаляет/обезличивает данные по каноническим правилам
* пишет результат
* завершает job
* ставит пользователю статус DELETED

9. Правила удаления по доменным блокам
9.1. Аккаунт и профиль
Нужно:
* запретить вход
* удалить/очистить:
    * nickname
    * full name
    * phone
    * email
    * bio
    * avatar url
    * birthday/age если есть
    * любые private preferences, если не нужны для compliance
* заменить публичное отображение на безопасный stub:
    * “Удалённый пользователь”
Важно: если nickname уже засветился в чужих историях, нельзя оставлять его как действующие PII; нужно заменить на stub в публичном отображении.

9.2. Auth и сессии
Нужно:
* инвалидировать refresh/access сессии
* удалить push tokens / device registrations
* запретить повторный логин в этот аккаунт
* если Supabase Auth хранит email/phone отдельно — предусмотреть канонический способ деактивации/удаления auth identity

9.3. Друзья / социальный граф
Нужно:
* удалить связи дружбы
* удалить/закрыть pending invites в друзья
* убрать пользователя из списков, рекомендаций, поисковых индексов

9.4. Планы
Нужно различать:
A. Пользователь — участник чужих планов
* удалить/деактивировать его членство там, где это допустимо
* в завершённых планах не ломать историю
* отображать как “Удалённый пользователь” там, где нужен исторический след
B. Пользователь — создатель плана
Нужно определить продуктовый контракт.
Рекомендованно:
* активные планы, где пользователь — owner:
    * либо автоматически архивировать/закрывать
    * либо передавать ownership нельзя без отдельной продуктовой механики
    * simplest V1: помечать план недоступным для дальнейшего редактирования и выводить участникам каноничное состояние
* завершённые планы:
    * сохранять как историческую сущность
    * owner в UI обезличивать
Это надо зафиксировать отдельно в продукте. Мой совет для V1: не удалять завершённые планы физически.

9.5. Голосования
Если голоса уже влияют на итог исторического плана:
* не удалять так, чтобы сломать итоговую историю
* обезличивать автора голоса при необходимости
* в активных ещё не завершённых сценариях можно удалить голос пользователя, если это не ломает бизнес-контракт — но это надо явно решить
Рекомендация V1:
* для завершённых планов сохранять голосовые факты как исторические записи без PII-display
* для незавершённых планов удалённого пользователя — удалять его участие из активного расчёта

9.6. Инвайты в планы
Нужно:
* удалить/закрыть все активные invite-ссылки пользователя как owner/inviter
* в исторических уже реализованных invite-flow оставить минимально нужный след без PII

9.7. Чаты и сообщения
Это один из главных блоков.
Рекомендация:
* не удалять физически все сообщения, иначе ты повредишь переписку других пользователей
* сообщения пользователя:
    * оставлять в цепочке
    * обезличивать автора
    * скрывать/очищать avatar/name
* если есть медиа-вложения в сообщениях:
    * private media удалять физически
    * в сообщении оставлять “Вложение удалено” или аналогичный stub
Это типовой и безопасный для продукта путь.

9.8. Пользовательские места
Если пользователь добавил место:
* само место, если оно стало частью общей базы, не удалять автоматически
* авторство места обезличить
* если место было только черновиком / submission без принятия — можно удалить физически
* если место прошло модерацию и стало частью каталога, лучше оставить объект, иначе пострадает продукт для всех пользователей
Это ключевой момент: место уже стало частью общего продукта, а не только личным файлом пользователя.

9.9. Медиа-файлы
Нужно:
* найти все user-owned media
* физически удалить из storage те файлы, которые:
    * относятся только к профилю пользователя
    * относятся к удаляемому приватному контенту
* для shared/public/media-in-history:
    * либо удалить и заменить заглушкой
    * либо обезличить ссылки на автора
Нужен чёткий media ownership contract.

9.10. Уведомления
Нужно:
* удалить pending notifications
* очистить device delivery records, push tokens
* historical notification logs — оставить при необходимости без PII

9.11. Бонусная система
Так как у тебя будет бонусный ledger и далее возможны покупки:
Нельзя просто физически удалять ledger
Правильнее:
* закрыть бонусный аккаунт
* current balance = 0 или account closed — решить продуктово
* ledger entries оставить как audit/history, но отвязать от публичного профиля и убрать display PII
* если будут реальные покупки/платежи, этот слой вообще нельзя стирать бесконтрольно
Рекомендация V1:
* при удалении аккаунта бонусный баланс аннулировать
* аккаунт бонусов закрыть
* ledger сохранить в системном контуре без публичного использования

9.12. Legal acceptance
Оставить записи о принятии документов в audit/legal контуре, если они нужны для споров и compliance. Публично не показывать.

9.13. Платежи и подписки
Сразу заложить этот блок, даже если он ещё не реализован.
Если появятся:
* подписки
* покупки
* пополнение баланса
то удаление аккаунта не должно:
* ломать платёжную историю
* стирать учёт обязательств, возвратов, chargeback-споров
* стирать налогово/учётно значимые записи
Поэтому для будущего блока payments нужно заранее заложить:
* retention policy
* отдельную таблицу правил хранения
* невозможность полного hard delete этого контура без специальных оснований

10. Ограничения перед удалением
Нужно проверить preconditions.
Минимум:
* нет ли незавершённых критичных операций удаления
* нет ли активной подписки/платёжного обязательства, которое требует отдельного UX предупреждения
* нет ли активного процесса модерации или abuse case, где аккаунт нельзя удалить мгновенно без server note
* пользователь действительно подтверждён и это его аккаунт
Важно: наличие таких сущностей не обязательно должно запрещать удаление, но может требовать:
* отдельного текста предупреждения
* иного server-flow
* отдельной retention policy

11. Канонический сценарий удаления
Шаг 1. Пользователь нажимает “Удалить аккаунт”
Клиент открывает экран подтверждения.
Шаг 2. Пользователь подтверждает
Клиент вызывает RPC request_account_deletion_v1.
Шаг 3. Сервер
* валидирует пользователя
* пишет audit
* переводит аккаунт в PENDING_DELETION
* создаёт job
* инвалидирует доступ
Шаг 4. Клиент
* очищает локальные данные
* разлогинивает
* уводит на стартовый экран
Шаг 5. Фоновый job processor
* выполняет шаги удаления/обезличивания
* пишет step logs
* при успехе ставит DELETED

12. Требования к идемпотентности
Удаление аккаунта должно быть идемпотентным.
Повторный вызов:
* не должен ломать состояние
* не должен повторно создавать одинаковые побочные эффекты
* должен корректно отвечать, что аккаунт уже в состоянии удаления/удалён
Нужны:
* unique constraint на active deletion job per user
* guard в RPC
* safe re-run processing шагов

13. RLS и безопасность
Пользователь может удалить только свой аккаунт.
Требования:
* клиент не может читать служебные deletion tables произвольно
* RPC работает через server-side trusted path
* все SQL-функции валидируют app.current_app_user_id
* admin/system-пути отделены от user-пути

18. Тест-кейсы
Обязательно проверить:
Базовый
* пользователь удаляет аккаунт
* вход блокируется
* профиль больше не виден
* job успешно завершается
Планы
* удаление участника не ломает завершённый план
* удаление owner не ломает историю
* активные сценарии обрабатываются канонично
Чаты
* переписка у собеседника остаётся
* автор обезличен
* приватные вложения удалены
Места
* одобренное место остаётся в базе
* автор обезличен
Бонусы
* бонусный аккаунт закрыт
* публичный баланс недоступен
* ledger сохранён как системная история
Идемпотентность
* повторный запрос не ломает систему
Безопасность
* нельзя удалить чужой аккаунт
* нельзя напрямую руками изменить deletion status с клиента

19. Рекомендуемое итоговое решение
Для Centry я рекомендую зафиксировать такой канон:
Делать не полное физическое удаление всего,
а гибрид:
* PII и приватные данные — удалять
* общепродуктовые исторические сущности — обезличивать
* юридически/финансово значимые записи — сохранять по retention policy
Это самый устойчивый вариант для продукта твоего класса.

20. Краткая постановка задачи для кодера
Нужно реализовать server-first сценарий удаления аккаунта пользователя в Centry с кнопкой в профиле/настройках, подтверждением на клиенте, серверным переводом аккаунта в pending deletion, фоновым job-процессингом, удалением персональных данных, обезличиванием исторических продуктовых записей, сохранением минимально необходимых системных/аудиторских/будущих платёжных следов, полной идемпотентностью и без разрушения целостности планов, чатов, мест и бонусной системы. Требуется предварительное исследование текущей схемы и выпуск карты затрагиваемых таблиц/функций до реализации.

Задачу выполнять канонически аккуратно, без слома текущих механик без даградции фунцианалаи и механик.
