ТЗ: Ревью и доработка базы знаний чата поддержки

# Цель

Сделать AI-бота поддержки Centry умнее и точнее. Пройтись по всем трём документам базы знаний, выявить пробелы, уточнить информацию у владельца продукта и дополнить БЗ до состояния, когда бот отвечает корректно на любой типовой вопрос пользователя.

# Контекст

Чат поддержки уже реализован и работает:
- Edge Function `support-send-question` — AI-чат с RAG (semantic search по pgvector + LLM через OpenRouter)
- Edge Function `kb-ingest` — индексация документов БЗ (chunking + embeddings)
- 3 документа БЗ загружены и проиндексированы (60 чанков)
- Flutter UI готов: экран выбора направления, AI-чат, формы предложений/жалоб
- Точка входа: Настройки → иконка поддержки

Проблема: бот иногда отвечает неточно или неполно, потому что БЗ содержит информацию в формате техдокументации для разработчика, а не в формате ответов для пользователя. Некоторые пользовательские сценарии не раскрыты или раскрыты частично.

# Документы БЗ

1. **BZ.md** — Продуктовая база знаний (~45K символов, 37 чанков). Полное описание продукта: сущности, сценарии, lifecycle, UI-экраны, пограничные случаи.
2. **Pravila_chat.md** — Правила поведения AI-бота (~11K символов, 9 чанков). Тон, границы компетенции, эскалация, prompt injection защита.
3. **SupportFAQ.md** — FAQ и сценарии эскалации (~16K символов, 14 чанков). Канонические ответы на частые вопросы, матрица эскалации.

# Порядок работы

## Этап 1: Исследование проекта

Перед ревью БЗ необходимо самостоятельно исследовать:
- **Серверную часть** (Supabase): таблицы, RPC-функции, бизнес-логику — чтобы понимать реальные механики продукта
- **Клиентскую часть** (Flutter): экраны, навигацию, UI-элементы — чтобы знать что видит пользователь
- Сопоставить описанное в БЗ с реальным кодом — найти расхождения

Цель: сформировать собственное понимание продукта на основе кода, а не только на основе документации. Если что-то можно дополнить в БЗ на основе кода — дополнить без вопросов.

## Этап 2: Ревью BZ.md

Пройтись по каждому разделу BZ.md последовательно:
1. Прочитать раздел
2. Оценить: достаточно ли информации для бота, чтобы он мог ответить пользователю на вопрос по этой теме?
3. Если раздел неполный, непонятный или требует уточнений — задать конкретные вопросы владельцу продукта
4. На основе ответов — дополнить/переписать раздел
5. Перейти к следующему разделу

Особое внимание:
- Пользовательские сценарии (как сделать X? где найти Y? что значит Z?)
- Ограничения и пограничные случаи
- Расхождения между описанием и реальным кодом
- Информация, написанная для разработчика, но бесполезная для бота поддержки

## Этап 3: Ревью Pravila_chat.md

Проверить:
- Актуальность правил поведения бота
- Полноту типовых ответов
- Корректность примеров
- Нет ли противоречий с BZ.md

## Этап 4: Ревью SupportFAQ.md

Проверить:
- Все ли частые вопросы покрыты
- Корректность канонических ответов
- Полноту сценариев эскалации
- Добавить новые FAQ на основе реальных механик из кода

## Этап 5: Переиндексация

После завершения всех правок:
1. Обновить контент документов в БД (kb_document_versions)
2. Запустить kb-ingest для каждого документа
3. Проверить количество и качество чанков
4. Протестировать бота на типовых вопросах
