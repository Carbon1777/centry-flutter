ТЗ: полноценный профиль пользователя в Centry
1. Контекст проекта

Работаем над проектом Centry — мобильным приложением на Flutter с backend на Supabase / Postgres / RPC / RLS / Realtime.

Архитектурный канон:

server-first строго

сервер — источник истины по продуктовым фактам

клиент — экран, кнопки, ввод, рендер

клиент не додумывает продуктовую логику локально

все профильные данные, настройки видимости, тип аватара, метаданные профиля — серверные

Продуктовая идентичность:

всё строится вокруг app_user_id

не строим продуктовую модель вокруг auth.uid()

2. Цель этапа

Нужно создать полноценный профиль пользователя.

Сейчас реально есть только:

ник

public id

email

Нужно добавить в рабочем виде:

имя

пол

возраст

аватар

настройки приватности

Также нужно подготовить и отрисовать будущие блоки:

Описание

Мои фото

Мои видео

Но эти три блока пока не функциональны, только как зарезервированное место под будущее развитие.

3. Форматы профиля
3.1. Полный профиль владельца

Полный экран профиля доступен только самому пользователю.

На этом экране он:

видит свои данные

редактирует:

имя

пол

возраст

аватар

переходит в настройки приватности

3.2. Короткая карточка чужого профиля

Короткая карточка открывается по тапу на ник / аватар в местах, где пользователь встречается в продукте:

друзья

планы

активности вокруг мест

лента

Именно на эту карточку влияют настройки приватности.

4. Что уже есть / что не трогаем
Уже есть и пока не редактируем:

ник

вводится при онбординге

в этом этапе не редактируется

public id

задаётся сервером

не редактируется

email

показывается только владельцу

не участвует в приватности

5. Новые редактируемые поля профиля
5.1. Имя

поле необязательное

можно оставить пустым

если не заполнено:

в полном профиле показываем прочерк

в короткой карточке тоже показываем прочерк, если поле разрешено по приватности

5.2. Пол

Значения строго фиксированные:

Мужской

Женский

Не указан

Если не заполнено:

в полном профиле — прочерк

в карточке — прочерк, если поле разрешено

5.3. Возраст

хранится как целое число лет

поле необязательное

диапазон пока не ограничиваем

ограничение только по формату:

максимум 2 цифры

если не заполнено:

прочерк в полном профиле

прочерк в карточке, если поле разрешено

6. Сохранение полей

Для:

имени

пола

возраста

аватара

сохранение происходит сразу, без общей кнопки “Сохранить профиль”.

То есть:

пользователь меняет значение

значение сразу уходит на сервер

UI обновляется от серверного ответа / серверного refresh

7. Аватар
7.1. Общая модель

У аватара два типа:

system

custom

Тип аватара — серверный факт.

7.2. Системные аватары

Есть заранее подготовленный набор системных аватаров.

Нужно:

положить системные аватары на сервер

сделать их доступными для выбора пользователем

отрисовать сетку выбора

Поведение:

пользователь открывает выбор аватара

видит сетку системных аватаров

выбирает один

выбор сохраняется сразу

7.3. Пользовательский аватар

Пользователь может загрузить свой аватар.

Важно:

это тоже серверный аватар

он должен быть виден другим пользователям, если это разрешено настройками приватности

локальным-only этот сценарий не делаем

7.4. Разрешения

Разрешение на доступ к фото/видео:

не запрашиваем заранее

запрашиваем только в момент, когда пользователь выбирает загрузить свой аватар

7.5. Кадрирование

После выбора фото открывается экран подгонки:

есть рамка аватара

пользователь может:

двигать изображение

масштабировать

сохраняется именно тот кадр, который пользователь оставил в рамке

Поведение как в мессенджерах.

7.6. Сохранение аватара

системный аватар сохраняется сразу

custom-аватар после кадрирования и подтверждения сохраняется сразу

общей кнопки сохранения профиля для этого нет

9. Настройки приватности
9.1. Вход в настройки

На полном экране профиля делаем вход в настройки приватности:

под кнопкой “Мои места”

в виде кликабельной строки / текста в стиле проекта

позже при необходимости можно перенести

9.2. Экран приватности

На отдельном экране пока делаем один блок:

блок настроек приватности

9.3. Модель приватности

Приватность делаем как независимую матрицу:

без наследования

без приоритетов

каждый переключатель живёт сам по себе

Контексты:

все

друзья

в планах

в ленте

Поля:

ник

аватар

имя

пол

возраст

9.4. Дефолт

По умолчанию:

всё всем показываем

9.5. Смысл контекстов

все — любой пользователь приложения

друзья — пользователи, которые состоят в друзьях

в планах — контекст планов и связанных активностей

в ленте — контекст ленты и активностей вокруг мест

10. Поведение короткой карточки чужого профиля

По тапу на ник / аватар в:

друзьях

планах

ленте

активностях вокруг мест

открывается короткая карточка профиля.

Если поле разрешено по приватности:

показываем реальное значение

Если поле скрыто:

показываем Скрыто

Специальные случаи:

если скрыт ник:

вместо ника показываем Пользователь

если скрыт аватар:

вместо аватара/фото показываем заглушку Скрыто

если скрыты имя / пол / возраст:

показываем Скрыто

Если поле не скрыто, но не заполнено:

показываем прочерк

11. Что показывает полный профиль владельца

В полном профиле владелец всегда видит свои данные полностью, независимо от приватности.

Если данные не заполнены:

имя → прочерк

пол → прочерк

возраст → прочерк

При этом рядом сохраняется возможность:

заполнить

поменять

выбрать другое значение

12. Server-first контракт профиля

Сервер должен быть источником истины для:

имени

пола

возраста

типа аватара

ссылки/идентификатора аватара

набора системных аватаров

настроек приватности

проекции короткой карточки чужого профиля под конкретный контекст

Клиент:

рендерит полный профиль владельца

рендерит короткую карточку другого пользователя

вызывает серверные методы

не считает правила видимости сам

13. Что должно быть на сервере

Нужно спроектировать и реализовать серверную основу для:

13.1. Профиль пользователя

Новые серверные поля:

name

gender

age

avatar_type

avatar_ref / avatar_url / avatar_asset_id

служебные поля по аватару, если нужно

13.2. Системные аватары

Нужно хранить:

список системных аватаров

идентификатор

публичный путь / url

порядок / сортировку

активность

13.3. Приватность

Нужна server-side модель матрицы:

контекст × поле

булевы настройки видимости

дефолт всё = true

13.4. Короткая карточка профиля

Сервер должен уметь отдавать карточку другого пользователя уже в готовом виде:

nickname/display nickname

avatar or hidden placeholder state

name or hidden/empty state

gender or hidden/empty state

age or hidden/empty state

17. Первый практический шаг

Так как начинать будем с аватарки, первый инженерный шаг должен быть исследовательским:

посмотреть текущие клиентские файлы профиля

понять, где уже выделено место под:

аватар

базовые поля

заглушки

посмотреть текущую серверную модель пользователя

понять, есть ли уже bucket / storage / media-инфраструктура, которую можно использовать для системных и пользовательских аватаров
