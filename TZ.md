1. Контекст проекта

Работаем над Centry — мобильным приложением на Flutter с backend на Supabase / Postgres / RPC / Realtime.

Архитектурный канон проекта:

server-first строго,

сервер — источник истины по продуктовым фактам,

клиент — максимально тупой UI,

клиент не додумывает бизнес-логику локально,

любые права, статусы, видимость, unread, message-state должны по возможности приходить с сервера.

Продуктовая идентичность:

всё строится вокруг app_user_id,

не строим продуктовую модель вокруг auth.uid().

2. Текущее фактическое состояние
2.1. Что уже готово на сервере

Серверная часть уже подготовлена для одиночных действий над сообщениями:

Уже есть:

edit_plan_chat_message_v1(...)

delete_plan_chat_message_for_me_v1(...)

delete_plan_chat_message_for_all_v1(...)

Уже поддерживается в snapshot:

edited_at

message_kind

deleted_at

персональные скрытия через plan_chat_message_member_hides

tombstone-логика для Удалить у всех

Семантика уже заложена правильно:

Редактировать → обновление текста + edited_at

Удалить у себя → персональное скрытие только для текущего app_user_id

Удалить у всех → не физическое удаление, а tombstone в истории

room_seq сохраняется

сервер сам должен считать видимое состояние сообщения

2.2. Что не готово

массовые действия (Выбрать несколько, чекбоксы, массовое delete-for-me) не реализованы

их не трогаем в первом восстановительном цикле

3. Цель текущего этапа

Нужно заново и аккуратно встроить в стабильную клиентскую чатовую базу три одиночных действия:

Редактировать

Удалить у себя

Удалить у всех

Без разрушения:

нижнего якоря списка,

поведения клавиатуры,

unread divider,

открытия/закрытия чата,

плавности,

уже утверждённого визуала.

4. Продуктовый контракт сообщений
4.1. Действия по long press
Для своего обычного сообщения:

Редактировать

Удалить у себя

Удалить у всех

Для чужого обычного сообщения:

Удалить у себя

Для tombstone-сообщения (Сообщение удалено):

Удалить у себя

нельзя:

редактировать

удалять у всех повторно

5. UX-контракт одиночных действий
5.1. Выделение сообщения

long press по bubble выделяет сообщение

выделение должно быть визуально аккуратным, без ломки геометрии

одновременно выделено только одно сообщение

tap по пустой области снимает выделение

5.2. Меню действий

после long press открывается bottom sheet / action sheet

пока меню открыто, сообщение остаётся выделенным

после закрытия меню выделение снимается

состав меню зависит от типа сообщения и прав

6. Контракт Редактировать
Поведение:

long press по своему сообщению

нажать Редактировать

меню закрывается

текст сообщения подставляется в composer

над composer появляется режим:

Редактирование сообщения

кнопка Отмена

клавиатура активна

отправка делает edit RPC

после успеха:

сообщение обновляется на месте

появляется метка Изменено

если текст не изменился:

можно либо ничего не делать,

либо просто выйти из edit mode

Ограничения:

tombstone нельзя редактировать

чужое сообщение нельзя редактировать

7. Контракт Удалить у себя
Поведение:

можно вызвать для своего и чужого сообщения

после действия сообщение исчезает только у текущего пользователя

у остальных ничего не меняется

последовательность истории не ломается

room order не ломается

Важно:

после удаления список не должен проседать под composer

нижний якорь чата должен оставаться корректным

если пользователь был у низа, низ должен остаться видимым

8. Контракт Удалить у всех
Поведение:

доступно только для своего обычного сообщения

после действия сообщение:

не исчезает физически,

остаётся на месте в истории,

превращается в tombstone

Как должен выглядеть tombstone:

текст: Сообщение удалено

более приглушённый визуальный стиль

без метки Изменено

без возможности:

редактировать

удалять у всех повторно

Важно:

Удалить у всех не должно превращаться в Изменено

не должно ломать scroll-position

не должно уводить нижнее сообщение под composer

9. Что не входит в этот этап

Пока не реализуем:

Выбрать несколько

чекбоксы слева у всех сообщений

массовое Удалить у себя

массовое Удалить у всех

массовое редактирование

reply / forward / pin

Это отдельный следующий этап.

10. Массовые действия — зафиксировать как следующий этап

Когда одиночные действия будут стабильно встроены, следующим этапом делаем:

Выбрать несколько

появляется пункт в action sheet: Выбрать несколько

чат входит в режим multi-select

слева у сообщений появляются чекбоксы

tap по сообщению или чекбоксу переключает выбор

появляется отдельная action bar

Для multi-select v1:

только Удалить у себя

без массового Удалить у всех

без редактирования

Это правильный и безопасный следующий шаг, но не сейчас.

11. Клиентская стратегия реализации

С учётом того, что сервер уже готов, клиент нужно восстанавливать строго снизу вверх, без прыжков.

Порядок встраивания:
Этап A — базовая совместимость контрактов

Сначала приводим в совместимость 4 файла:

plan_chat_dto.dart

plan_chat_block.dart

plan_chat_sheet.dart

plan_chat_message_bubble.dart

Чтобы они все одинаково понимали:

editedAt

messageKind

deletedAt

Цель этапа:

нулевые красные ошибки

нулевые регрессы UX

Этап B — Удалить у себя

Потом встраиваем только:

action sheet

callback

вызов repository

refresh snapshot

И отдельно проверяем:

низ списка не уезжает под composer

Этап C — Редактировать

После этого:

edit mode

banner над composer

send → edit RPC

метка Изменено

Этап D — Удалить у всех

После этого:

пункт Удалить у всех

tombstone в UI

без регресса на scroll / bottom anchor

12. Критерии готовности этапа

Этап можно считать успешно завершённым, если:

Редактирование:

работает только для своего обычного сообщения

текст подставляется в composer

после сохранения сообщение обновляется на месте

есть метка Изменено

Удалить у себя:

работает для своего и чужого

сообщение исчезает только у меня

низ списка не проваливается под composer

Удалить у всех:

работает только для своего обычного сообщения

сообщение остаётся на месте

превращается в Сообщение удалено

не показывает Изменено

для него больше нельзя вызвать Редактировать и Удалить у всех

Поведение чата:

не сломан unread divider

не сломан keyboard behavior

не сломан open/close sheet

не сломан нижний якорь списка

не появилось новых дерганий

13. Правила работы в новой сессии / продолжении
