ТЗ для создания размещения и реализации соглашений и правил в проекте Задача создать и реализовать правильную систему фиксация полученных соглашений от пользователей дабы исключить возможные юридические риски. Внедрить систему в проект и опубликовать документы в проекте. Сервер фикс, сервер отдает, клиент рисует.  Если будут советы по улучшению функционала и самих токов - приветствуется.  Как правильно фиксировать принятие пользовательского соглашения
Это очень важный момент. 90% стартапов делают это неправильно.
Правильная схема выглядит так:
UI галочка ≠ юридическое принятие. Юридическим доказательством является серверная запись факта согласия.

1. Что должно фиксироваться на сервере
Нужно сохранять:
* app_user_id
* версия соглашения
* версия политики данных
* версия правил бонусов
* дата и время принятия
* ip пользователя
* user agent
* версия приложения
Это стандартная практика.

2. Таблица принятия соглашения
Создать таблицу:

user_legal_acceptances

Пример структуры:

id (uuid)
app_user_id (uuid)

terms_version
privacy_version
bonus_rules_version

accepted_at (timestamp)

ip_address
user_agent
app_version

created_at

Важно:
* immutable записи
* нельзя редактировать
* только добавлять новые

3. Таблица документов

legal_documents


id
document_type

version
title
content

published_at
is_active

document_type:

TERMS
PRIVACY
BONUS_RULES


4. Правильный flow принятия
Шаг 1
Клиент запрашивает:

get_current_legal_documents()

Сервер отдаёт:

terms_version
privacy_version
bonus_rules_version
content


Шаг 2
Клиент показывает экран:
☑ Я принимаю пользовательское соглашение ☑ Я принимаю политику обработки данных ☑ Я принимаю правила проекта

Все строки кликабельные (ссылки) ведут каждая в свой документ (экран) Ссылка единственная точка входа в экран. 

Кнопка Продолжить
 Клиент только рисует Сами текста находятся на сервере и отдаются в клиент.  
Шаг 3
Клиент вызывает RPC:

accept_legal_documents

Параметры:

app_user_id
terms_version
privacy_version
bonus_rules_version


Шаг 4
Сервер записывает:

user_legal_acceptances


Шаг 5
Если пользователь не принял соглашение и правила — приложение не работает - не сохраняем ник не пускаем дальше при клике пройти дальше если не стоит галочка на каком-то из трех пунктов выводим наше уведомление через централ тост какой пункт не проставлен пример: «Вы не приняли правила проекта».

5. Проверка при входе
При каждом входе сервер должен проверять:

accepted_terms_version == current_terms_version

Если версия изменилась:
показать экран обновлённого соглашения.
Это стандартная практика.

6. Почему это важно
Если когда-нибудь будет спор:
пользователь скажет:
"Я не принимал ваши правила".
Ты покажешь запись:

user_id
version
timestamp
ip
device

Настоящее Пользовательское соглашение (далее — «Соглашение») регулирует отношения между сервисом Centry (далее — «Сервис») и физическим лицом, использующим мобильное приложение Centry (далее — «Пользователь»).
Используя Сервис, Пользователь подтверждает, что ознакомился с настоящим Соглашением, Политикой обработки персональных данных и Правилами сообщества и принимает их в полном объёме.

1. Термины
Сервис — программный продукт Centry. Пользователь — лицо, использующее Сервис. Контент — любые данные, размещённые пользователем (текст, фото, видео, места, сообщения). Аккаунт — учётная запись пользователя.

2. Предмет соглашения
Сервис предоставляет Пользователю доступ к функциональности приложения, включая:
создание и управление планами
участие в голосованиях
обмен сообщениями
публикацию контента
добавление мест
социальное взаимодействие
Сервис не является стороной личных взаимодействий между пользователями.

3. Регистрация и аккаунт
Пользователь может предоставить:
никнейм
номер телефона
email
фото профиля
Пользователь обязуется:
предоставлять достоверные данные
не передавать доступ третьим лицам
не создавать аккаунты для злоупотребления
Сервис вправе ограничить доступ при нарушениях.

4. Пользовательский контент и лицензия
Пользователь сохраняет права на свой контент.
Размещая контент, Пользователь предоставляет Сервису неисключительную лицензию на:
хранение
отображение
обработку
использование внутри сервиса
Пользователь гарантирует:
наличие прав на контент
отсутствие нарушений законодательства

5. Ограничения
Запрещено:
нарушать закон
публиковать незаконный контент
распространять спам
оскорблять пользователей
обходить ограничения
использовать ботов
манипулировать бонусной системой

6. Мессенджер и взаимодействие
Пользователи могут обмениваться сообщениями.
Пользователь несёт ответственность за свои сообщения.
Сервис вправе:
ограничить доступ к сообщениям
удалить переписку при нарушениях

7. Бонусная система
Токены:
являются внутренней системой поощрения
не являются деньгами
не имеют гарантированной стоимости
Сервис вправе:
изменять правила начисления
ограничивать начисления
аннулировать токены при злоупотреблениях

8. Платные функции
Сервис может предоставлять:
подписки
платные функции
цифровые продукты
Платежи осуществляются через сторонние платформы (App Store / Google Play и др.).
Возвраты регулируются правилами этих платформ.

9. Модерация
Сервис вправе:
удалять контент
ограничивать функциональность
блокировать аккаунты
без предварительного уведомления.

10. Ограничение ответственности
Сервис не несёт ответственности за:
действия пользователей
пользовательский контент
личные встречи пользователей
возможный ущерб

11. Изменения
Сервис вправе изменять Соглашение.
Продолжение использования означает согласие.
