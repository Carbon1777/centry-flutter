# ТЗ: Feed (Лента) — полная спецификация

---

## 0. Архитектурный канон

- server-first строго — сервер источник истины, клиент тупой UI
- Клиент не додумывает бизнес-логику локально
- Все права, статусы, агрегаты, сигналы приходят с сервера
- Всё строится вокруг app_user_id, не вокруг auth.uid()

---

## 1. Что такое Feed и чем он не является

Feed — витрина мест, управляемая сервером. Отвечает на вопрос: "Какие места мне релевантны прямо сейчас в моём районе?"

Единица выдачи: Place (место)

- Plan НЕ является элементом Feed
- Plan влияет на Feed только через сигналы и агрегаты
- Feed НЕ даёт доступ к плану
- Feed НЕ даёт plan_id как путь входа в план
- Feed НЕ является точкой входа в чужой план
- Feed — это витрина, не дверь

## 2. Пайплайн формирования Feed (сервер)

### Шаг 1 — Контекст пользователя

Входные параметры от клиента:
- geo (lat/lng, accuracy)
- время "сейчас"
- user_id (если залогинен) или guest

Сервер вычисляет:
- area_id (район/зона) — динамически, не сохраняется в app_users
- радиусы расширения: R1 → R2 → R3 → до границ города

Инвариант: geo-фильтр — самый первый и жёсткий

### Шаг 2 — Набор кандидатов

- Базовые места района (core_places by area_id)
- Места с активностью (планы/сигналы) в районе
- Места с системной активностью (trending, editorial)

Расширение радиуса:
- Если в R1 меньше 25 мест → расширяем до R2
- Если в R2 меньше 25 мест → расширяем до R3
- R3 не предел — расширяем до границ города пока не наберём 25 мест
- Feed не должен быть пустым (кроме крайнего случая отсутствия мест в городе)

Целевой размер выдачи: 25 мест

### Шаг 3 — Агрегаты по месту

Плановые агрегаты:
- count_plans_by_place — кол-во активных планов вокруг места
- interested_count — уникальные голоса за место в активных планах (до финализации)
- planned_count — уникальные участники финализированных планов на это место
- visited_count — участники завершённых планов (позже, сейчас default 0)

Динамические:
- activity_velocity — скорость роста активности
- freshness — свежесть активности

Инвариант уникальности: один пользователь учитывается один раз на одно место, даже если у него несколько планов с этим местом.

### Шаг 4 — Рейтинг (score)

score(place) =
  Wp * plan_signal_score(place)
+ Ws * system_signal_score(place)
+ Wr * recency_velocity_score(place)
+ Wq * quality_baseline(place)
- D  * decay(place)

Где Wp >> Ws.

Plan-signal score:
  a * f(planned_count)       — наибольший вес
+ b * f(interested_count)    — средний вес
+ c * f(visited_count)       — ниже
+ d * f(count_plans_active)  — стабилизатор

f(x) — сглаживающая функция (логарифм или корень), чтобы популярные места не доминировали бесконечно.

Места в ленте выводятся по убыванию score — самые популярные вверху, менее популярные внизу.
Без секций и заголовков — один сплошной список.

## 3. Геолокация и поведение клиента

Сценарий 1 — геолокация доступна:
- Строим ленту по текущему гео

Сценарий 2 — текущего гео нет, есть последнее известное:
- Используем последнее известное гео

Сценарий 3 — геолокация никогда не предоставлялась:
- Строим ленту рандомно по рейтингу и активности
- Показываем уведомление

Уведомление при отсутствии геолокации:
- Текст: "Лента построена случайным образом, так как вы не предоставили доступ к геолокации. Мы можем персонализировать её специально для вас."
- Кнопка "Дать разрешение" → системный запрос геолокации
- Крестик (закрыть) → показываем снова при следующем запуске приложения с задержкой 1 сек после загрузки ленты
- Закрыть свайпом → поведение как у крестика
- Мелкая кнопка "Больше не показывать" → сохраняем флаг локально, никогда больше не показываем за весь цикл жизни приложения на устройстве
- Если геолокация предоставлена — уведомление не показываем

## 4. Обновление ленты

- Лента загружается заново при каждом входе на экран Feed
- Pull-to-refresh: потянуть вниз → перезагрузка
- Realtime и push-уведомления — отдельный этап, не сейчас

## 5. Read-модель (серверные таблицы)

Перед реализацией Claude Code исследует текущее состояние базы и проверяет что уже существует.

### 5.1. Агрегаты по месту

Таблица: plan_public_aggregates_by_place

Колонки:
- place_id (PK)
- count_plans_by_place (int)
- interested_count (int)
- planned_count (int)
- visited_count (int, default 0, реализуется позже)
- updated_at (timestamp)

### 5.2. Оболочки планов по месту

Таблица: plan_public_shells_by_place

Ключ: (place_id, plan_id)

Колонки:
- place_id
- plan_id — opaque id, не даёт доступа в план
- is_visible (bool) — настройка owner'а плана
- title (text, nullable) — только если is_visible = true
- participants_count (int) — всегда
- participants_public_preview (jsonb, nullable) — только если is_visible = true
- signal_phase (enum: INTERESTED / PLANNED / VISITED)
- updated_at (timestamp)

participants_public_preview — массив до 5 элементов:
  { "kind": "public", "nickname": "...", "avatar_url": "...", "public_id": "..." }
  { "kind": "anon" }  — если профиль участника приватный

### 5.3. Что строго исключено из plan_public_shells

- кандидатные списки
- голоса
- даты и дедлайны
- комментарии
- роли участников
- invite-токены
- любые ссылки на чат

## 6. Семантика сигналов

INTERESTED (Интересуются):
- Источник: план не финализирован, место является кандидатом
- Каждый участник плана даёт 1 уникальный сигнал на место
- Дедуп: один user = один сигнал на место
- После финализации плана interested_count перестаёт расти

PLANNED (Идут):
- Источник: финализация плана, decided_place_id = place_id
- Каждый участник даёт 1 уникальный сигнал
- После финализации interested перетекает в planned

VISITED (Были):
- Источник: план завершён, event_at < now()
- Реализуется позже, сейчас visited_count = 0

## 7. Триггерная модель пересчёта (без cron)

Что триггерит пересчёт:
1. Membership: join / leave / remove member
2. Places candidates: add / remove candidate
3. Place votes: insert / update / delete
4. Finalization: finalize_plan_v1, resolve_tie_v1
5. Plan visibility / title: обновление
6. Plan deletion: owner delete / system delete

Каноническая процедура:

refresh_place_plan_public_read_model(place_id):
- пересчитывает агрегаты для place
- пересобирает список оболочек для place

refresh_places_for_plan(plan_id, old_row?, new_row?):
- вычисляет затронутые place_id
- вызывает refresh_place_plan_public_read_model для каждого

Как вычислять затронутые place_id:
- все place_id из core_plan_place_candidates для plan_id
- decided_place_id (старое и новое, если изменилось)
- при удалении кандидата: учитывать и удалённый place_id
- при удалении плана: все кандидаты + decided_place_id

Ограничение 5 кандидатов → максимум ~6 place_id за событие → дёшево.

Где вызываем:
- Внутри RPC после успешной мутации — основной путь
- Триггеры на таблицах — safety net

## 8. Инварианты безопасности

Feed читает ТОЛЬКО:
- plan_public_aggregates_by_place
- plan_public_shells_by_place

На этих таблицах:
- RLS: read разрешён всем (данные уже очищены на уровне структуры)
- Write только через SECURITY DEFINER функции / service role

## 9. RPC для клиента

get_feed_nearby(lat, lng, limit):
- Возвращает список мест с агрегатами и score
- Отсортирован по убыванию score
- limit = 25

get_place_plan_shells(place_id):
- Возвращает список оболочек планов для места
- Только активные планы (не завершённые)
- Сортировка: случайная

- Не кликабельно

## 18. Порядок реализации

Этап 1 — Исследование (обязательно первым):
- Исследовать текущее состояние базы
- Проверить наличие таблиц plan_public_aggregates_by_place и plan_public_shells_by_place
- Проверить существующие RPC и триггеры связанные с Feed
- Проверить текущий activity_feed_screen.dart
- Никаких правок до завершения исследования

Этап 2 — Сервер: read-модель:
- Создать таблицы агрегатов и оболочек (если не существуют)
- Создать процедуры refresh_place_plan_public_read_model и refresh_places_for_plan
- Подключить вызовы refresh внутри существующих RPC

Этап 3 — Сервер: RPC для клиента:
- get_feed_nearby(lat, lng, limit)
- get_place_plan_shells(place_id)

Этап 4 — Клиент: DTO и Repository:
- FeedPlaceDto, PlanShellDto, ParticipantPreviewDto
- FeedRepository, FeedRepositoryImpl

Этап 5 — Клиент: экран и карточка:
- activity_feed_screen.dart
- feed_place_card.dart

Этап 6 — Клиент: детали и модалки:
- feed_place_detail_sheet.dart
- feed_plan_shells_modal.dart
- feed_plan_participants_modal.dart

Этап 7 — Клиент: геолокация и уведомление:
- Логика определения гео (текущее → последнее → рандом)
- feed_geo_banner.dart с тремя кнопками и локальным флагом

## 19. Что НЕ входит в этот этап

- Realtime обновления и push-уведомления
- visited_count / сигнал VISITED — позже
- Коммерческие и партнёрские override блоки
- Аватарки участников прямо на карточке в ленте
- Пагинация (25 мест достаточно)

## 20. Критерии готовности

Сервер:
- Таблицы read-модели существуют и заполняются корректно
- Пересчёт агрегатов срабатывает после каждого доменного события
- get_feed_nearby возвращает корректно отсортированный список
- get_place_plan_shells возвращает только активные планы

Клиент:
- Лента загружается, показывает места с актуальными сигналами
- Карточка содержит: фото, название, категорию, расстояние, метро, рейтинг, счётчик планов, сигналы
- Клик на карточку → bottom sheet деталей
- Все существующие механики в деталях работают без регрессов
- Кнопка "Планов — N" → модалка списка планов
- Видимый план → клик на участников → модалка участников
- Клик на публичного участника → существующая карточка профиля
- Приватный план: "План скрыт · N участников", не кликабельно
- Приватный участник: иконка + "Скрыто", не кликабельно
- Pull-to-refresh работает
- Геолокация: все три сценария обработаны
- Уведомление о геолокации: крестик / дать разрешение / больше не показывать
- Спиннер + текст при загрузке
- Пустое состояние при отсутствии мест
- Существующий UI других экранов не сломан
