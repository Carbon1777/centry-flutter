## 1. ОБЩЕЕ ОПИСАНИЕ ПРОДУКТА

### Что это за приложение

**Centry** — мобильное приложение для совместного планирования досуга с друзьями. Пользователи находят интересные места (рестораны, бары, кинотеатры и др.), создают планы-встречи, голосуют за дату и место, общаются в чатах и обмениваются знаками внимания.

### Основная ценность

Centry решает проблему «куда пойти и когда» для компании друзей. Вместо бесконечной переписки «а давай сюда? а когда?» приложение даёт структурированный процесс: создать план → предложить варианты мест и дат → проголосовать → встретиться.

### Ключевые пользовательские задачи

1. **Найти интересное место** — просмотр каталога мест с фильтрами, картой, рейтингами
2. **Организовать встречу** — создание плана с голосованием по датам и местам
3. **Пригласить друзей** — через инвайт-ссылки, из списка друзей или по публичному ID
4. **Обсудить план** — встроенный чат внутри каждого плана
5. **Общаться лично** — приватные чаты между друзьями
6. **Показать симпатию** — ежедневная механика знаков внимания
7. **Отслеживать активность** — лидерборд по токенам и симпатиям

### Главные продуктовые сущности

| Сущность | Суть |
|----------|------|
| **Пользователь** | Участник приложения с профилем, предпочтениями и аватаром |
| **Место** | Заведение из каталога (ресторан, бар, кинотеатр и т.д.) |
| **План** | Событие/встреча с голосованием за дату и место |
| **Дружба** | Двусторонняя социальная связь между пользователями |
| **Приватный чат** | Переписка между двумя пользователями |
| **Знак внимания** | Ежедневная отправка символа симпатии другу |
| **Бонусные токены** | Виртуальная валюта за активность |
| **Блокировка** | Механизм защиты от нежелательных контактов |

## 2. КАРТА ПРОДУКТА

### 2.1 Онбординг и вход

**Как пользователь попадает:** первое открытие приложения.

**Шаги:**
1. **Экран никнейма** — ввод ника (2–20 символов), чтение и принятие трёх юридических документов (Пользовательское соглашение, Политика конфиденциальности, Правила проекта). После подтверждения на сервере создаётся пользователь в статусе GUEST.
2. **Экран разрешений** — запрос геолокации и уведомлений. Оба можно пропустить.
3. **Экран email** — ввод email (опционально, можно пропустить). При вводе: проверка доступности → отправка OTP-кода → подтверждение → статус меняется на USER.

**Результат:** пользователь может быть в статусе GUEST (только ник, без email) или USER (ник + email + авторизация).

**Возвращающиеся пользователи:** при следующем открытии приложение проверяет наличие авторизационной сессии или локального снимка данных и сразу ведёт на главный экран. Если версии юридических документов обновились — показывается экран переакцептации.

### 2.2 Главный экран (Приветствие → Лента)

**Как попадает:** после онбординга или авторизации.

**HomeScreen** показывает анимированное приветствие с ником пользователя (~5.5 сек), затем переходит к основной навигации.

### 2.3 Основная навигация (4 вкладки)

| Вкладка | Что делает | Бейдж |
|---------|-----------|-------|
| **Места** | Каталог мест: список + карта + фильтры | Нет |
| **Мои планы** | Список активных и архивных планов | Красная точка при непрочитанных сообщениях в чатах планов |
| **Друзья** | Список друзей + добавление по ID | Нет |
| **Чаты** | Список приватных переписок | Красная точка при непрочитанных сообщениях |

**Дополнительно в верхней панели:**
- Кнопка профиля (с красной точкой если есть входящие знаки внимания)

### 2.4 Места

**Как попадает:** вкладка «Места» в навигации.

**Для чего:** поиск и просмотр заведений для встреч.

**Функции:**
- Список мест с карточками (название, категория, фото, рейтинг, расстояние, метро)
- Карта OpenStreetMap с маркерами и кластеризацией
- Фильтры: город, район, категория (ресторан/бар/клуб/кино/театр/караоке/кальянная/баня), минимальный рейтинг
- Поиск по названию с автодополнением
- Сохранение места в «Мои места»
- Голосование (лайк/дизлайк) за место
- Добавление нового места (пользовательская подача)
- Добавление места в план

### 2.5 Мои планы

**Как попадает:** вкладка «Мои планы».

**Для чего:** управление встречами.

**Функции:**
- Две под-вкладки: Активные / Архив
- Создание нового плана
- Просмотр деталей плана (голосование, участники, чат)
- Приглашение участников

### 2.6 Друзья

**Как попадает:** вкладка «Друзья».

**Для чего:** управление социальными связями.

**Функции:**
- Список всех друзей с мини-профилем
- Добавление друга по публичному ID
- Отправка знака внимания (long press)
- Создание приватного чата с другом
- Добавление друга в план
- Заметки о друге (личные, видны только вам)
- Удаление из друзей

### 2.7 Чаты

**Как попадает:** вкладка «Чаты».

**Для чего:** личная переписка.

**Функции:**
- Список чатов с превью последнего сообщения и счётчиком непрочитанных
- Обмен текстовыми сообщениями
- Редактирование и удаление сообщений (для себя / для всех)
- Удаление чата / удаление чата с блокировкой партнёра

### 2.8 Профиль

**Как попадает:** иконка в правом верхнем углу навигации.

**Для чего:** управление своим аккаунтом.

**Подразделы:**
- Основная информация (ник, имя, пол, возраст, город, био)
- Аватар (фото, загрузка, кроп)
- Фото профиля (до нескольких фото с порядком)
- Предпочтения досуга (формат отдыха, компания, время, вайб)
- Настройки приватности (что видят другие в планах и ленте)
- Мои места (сохранённые)
- Лидерборд
- Centry Market (бонусы)
- Заблокированные пользователи
- Удаление аккаунта

### 2.9 Коробка знаков внимания

**Как попадает:** через профиль или по нажатию на бейдж.

**Для чего:** социальная механика симпатий.

**Содержит три секции:**
1. Мой знак на сегодня (можно отправить одному другу)
2. На рассмотрении (входящие знаки — принять / отклонить)
3. Коллекция (принятые знаки с счётчиками)

### 2.10 Лидерборд

**Как попадает:** через профиль.

**Две вкладки:**
- **Активность**: рейтинг по токенам + рейтинг по завершённым планам
- **Симпатии**: рейтинг полученных знаков + рейтинг отправленных

Каждый рейтинг показывает TOP-10 + текущее место пользователя.

### 2.11 Centry Market

**Как попадает:** через профиль (иконка в верхней части экрана профиля).

**Для чего:** просмотр баланса токенов, история начислений, правила начисления. Тратить токены пока нельзя — функция откроется с релизом.

### 2.12 Поддержка

**Как попадает:** Профиль → Настройки (иконка замка) → иконка вопроса в правом верхнем углу.

**Для чего:** связь с командой Centry.

**Три направления:**
1. **Вопросы** — AI-чат по базе знаний. Бот отвечает по контексту приложения, при отсутствии ответа предлагает создать заявку.
2. **Предложения** — форма с текстовым полем. После отправки: «Спасибо, ваше предложение принято и будет рассмотрено».
3. **Жалобы** — форма с текстовым полем. После отправки: «Спасибо, ваша жалоба принята в обработку».

### 2.13 Чат внутри плана

**Как попадает:** через детали плана (блок «Чат» внизу).

**Для чего:** обсуждение деталей встречи участниками плана.

**Функции:**
- Текстовые сообщения
- Редактирование и удаление сообщений (для себя / для всех)
- Бейдж непрочитанных на карточке плана и на вкладке «Мои планы»
- В архиве — только чтение

### 2.14 Юридические документы

При онбординге пользователь принимает 3 документа:
1. **Пользовательское соглашение**
2. **Политика конфиденциальности**
3. **Правила проекта**

Если версии документов обновляются — при следующем входе показывается экран обязательной переакцептации.

## 3. ОСНОВНЫЕ СУЩНОСТИ ПРОДУКТА

### 3.1 Пользователь

**Что это:** человек, зарегистрированный в приложении.

**Где возникает:** при онбординге (ввод ника).

**Состояния:**
- **GUEST** — пользователь с ником, но без подтверждённого email. Может просматривать места, но не может создавать планы.
- **USER** — полноценный пользователь с email. Может использовать все функции.

**Идентификация:**
- `app_user_id` — внутренний ID (основа всех сценариев, не auth-идентификатор)
- `public_id` — публичный уникальный код для обмена (можно дать другу для добавления)
- `nickname` — отображаемое имя

**Профиль включает:**
- Имя, пол, возраст, город, краткая биография
- Аватар: 60 системных аватаров на выбор или загрузка своего фото с кадрированием
- Фотоальбом: до 9 фото в бета-версии. Можно загружать, удалять, менять порядок перетаскиванием. Фото открываются в полноэкранном режиме с зумом и свайпами. С релизом появится возможность загружать больше фото и видео.
- Предпочтения досуга: формат отдыха, формат компании, темп встреч, время встреч, вайб
- Заметки о друзьях: личная пометка о друге, видна только автору заметки
- Настройки приватности

**Жизненный цикл:**
1. Создание (GUEST) → подтверждение email → USER
2. Активное использование
3. Удаление аккаунта (2-шаговый процесс: запрос → финализация)

### 3.2 Место (Place)

**Что это:** заведение или локация для проведения досуга.

**Где возникает:** загружается из серверного каталога (9400+ мест) или создаётся пользователем (submission).

**Свойства:**
- Название, тип/категория, адрес
- Координаты (широта, долгота)
- Ближайшее метро с расстоянием
- Рейтинг
- Превью-изображение
- Веб-сайт (если известен)

**Категории мест:**
- Ресторан, Бар, Ночной клуб, Кинотеатр, Театр, Караоке, Кальянная, Баня/Сауна

**География:**
Каталог работает в 8 городах:
- Москва (~4500 мест)
- Санкт-Петербург (~2900 мест)
- Новосибирск (~400 мест)
- Нижний Новгород (~370 мест)
- Ростов-на-Дону (~350 мест)
- Казань (~320 мест)
- Сочи (~310 мест)
- Краснодар (~270 мест)
Итого: 9400+ мест в каталоге.

**Рейтинг мест:** гибридная система — базовый рейтинг из внешних источников + лайки/дизлайки от пользователей Centry.

**Кто может:**
- Просматривать — все пользователи
- Голосовать (лайк/дизлайк) — все пользователи
- Сохранять в «Мои места» — все пользователи
- Добавлять новое место (submission) — все пользователи
- Добавить в план — участники плана

**Состояния пользовательского места (submission):**
- На модерации (pending)
- Одобрено
- Отклонено (rejected)

### 3.3 План (Plan)

**Что это:** событие/встреча, которую организует группа пользователей с голосованием за дату и место.

**Где возникает:** создаётся пользователем (статус USER обязателен).

**Свойства:**
- Название, описание (мин. 10 символов)
- Дедлайн голосования
- Видимость в ленте (on/off)
- Выбранная дата и место (результат голосования)
- Список участников

**Состояния:**
- **OPEN** — идёт голосование, можно добавлять кандидатов и голосовать
- **VOTING_FINISHED** — голосование завершено, определены победители
- **CLOSED** — событие состоялось, план в архиве

**Дополнительные флаги:**
- `archived` — переведён в архив пользователем (любой статус)
- `ownerChoiceModeActive` — ничья в голосовании, владелец должен выбрать приоритет

**Кто может создать:** только USER (не GUEST).

**Чем заканчивается:** статус CLOSED, план уходит в архив.

### 3.4 Дружба (Friendship)

**Что это:** двусторонняя социальная связь между двумя пользователями.

**Где возникает:** один пользователь отправляет запрос другому по публичному ID.

**Состояния запроса:**
- **PENDING (OUTGOING)** — запрос отправлен, ожидает ответа
- **PENDING (INCOMING)** — входящий запрос от другого пользователя
- **ALREADY_FRIENDS** — уже друзья

**Что на неё влияет:**
- Принятие запроса → дружба установлена
- Отклонение запроса → запрос удаляется
- Удаление друга → дружба разрывается
- Блокировка → дружба разрывается автоматически

**Жизненный цикл:**
1. Отправка запроса (PENDING)
2. Принятие / Отклонение
3. Активная дружба
4. Удаление или блокировка

### 3.5 Приватный чат

**Что это:** личная переписка между двумя пользователями.

**Где возникает:** создаётся из карточки друга (кнопка «Создать чат»).

**Предусловие:** `canCreateChat = true` (определяется сервером с учётом настроек приватности и блокировок).

**Важное правило:** дружба нужна для **создания** нового чата, но не для жизни уже существующего. Если дружба удалена без блокировки — чат остаётся живым, в него можно продолжать писать.

**Возможности:**
- Отправка текстовых сообщений и emoji
- Редактирование своих сообщений
- Удаление сообщений (для себя / для всех)
- Пагинация (подгрузка старых сообщений)
- Счётчик непрочитанных

**Чем заканчивается:** удаление чата (вручную) или блокировка (автоматически удаляет чат и историю).

### 3.6 Знак внимания (Attention Sign)

**Что это:** ежедневный механизм демонстрации симпатии.

**Где возникает:** каждый день в 00:00 UTC пользователю выдаётся один знак.

**Свойства знака:**
- Тип (6 типов, у каждого свой стикер/изображение):
  - Бриллиантовое ожерелье
  - Золотые часы
  - Бриллиантовое кольцо
  - Красный Ferrari
  - Букет роз
  - Бутылка виски
- Дата выдачи
- Срок действия (до 00:00 следующего дня)

**Откуда можно отправить знак (3 места):**
1. Список друзей — long press на карточку друга
2. Список участников плана (внутри плана) — long press на участника
3. Список участников плана в ленте мест — long press на участника

**Жизненный цикл:**
1. Выдача знака (00:00 UTC)
2. Пользователь отправляет знак другу (или не отправляет → знак пропадает)
3. Получатель видит знак во «Входящих»
4. Получатель принимает (→ знак в коллекцию) или отклоняет (→ знак исчезает)

**Особая механика:** при принятии знака от незнакомца — возможность использовать «право на приглашение в друзья» (отправить запрос дружбы атомарно).

### 3.7 Бонусные токены

**Что это:** виртуальная валюта, начисляемая за активность.

**Где возникает:** сервер начисляет автоматически при выполнении действий.

**Правила начисления (точные данные):**

| Действие | Токены | Описание |
|----------|--------|----------|
| Успешно завершённый план (владелец) | **10** | Ты создал план, который успешно завершился |
| Голосование за дату в завершённом плане | **1** | Ты проголосовал за дату |
| Голосование за место в завершённом плане | **1** | Ты проголосовал за место |
| Регистрация по приглашению в план | **15** | Кто-то зарегистрировался по твоему приглашению |
| Место прошло модерацию | **12** | Твоё добавленное место одобрено |
| Место стало местом встречи | **5** | Твоё место выбрано финальным в плане |

**Свойства:** текущий баланс, история операций.

**Трата токенов:** в текущей версии тратить токены нельзя. Centry Market с возможностью траты откроется с релизом. Все накопленные токены сохранятся.

### 3.8 Блокировка

**Что это:** механизм защиты от нежелательных контактов.

**Что происходит при блокировке (атомарно):**
- Разрывается дружба
- Удаляется приватный чат (если был)
- Создаётся запись о блокировке

**Подтверждено:** разблокировка в текущей версии не реализована ни на клиенте, ни на сервере. Планируется ближе к релизу.

## 4. ПОЛНЫЕ ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ

### 4.1 Онбординг нового пользователя

**Точка входа:** первый запуск приложения.
**Предусловия:** нет.

**Шаги:**
1. Пользователь видит экран ввода ника
2. Вводит ник (2–20 символов)
3. Читает и принимает три юридических документа (чекбоксы)
4. Нажимает «Далее» → на сервере создаётся GUEST-пользователь
5. Экран разрешений: запрос геолокации → можно разрешить или пропустить
6. Запрос уведомлений → можно разрешить или пропустить
7. Экран email: можно ввести email или пропустить
   - Если email введён: проверка доступности → OTP-код на email → подтверждение → статус USER
   - Если email занят: предлагается восстановление доступа
   - Если пропущен: остаётся GUEST
8. Анимация приветствия (HomeScreen) → переход к основной навигации

**Неуспешные исходы:**
- Сервер недоступен → retry с backoff (400ms)
- Email уже занят → предложение восстановления
- OTP-код неверен → повторный ввод

### 4.2 Создание плана и голосование

**Точка входа:** кнопка «Создать план» на вкладке «Мои планы».
**Предусловия:** пользователь в статусе USER (не GUEST).

**Шаги:**
1. Заполнение формы: название, описание (мин. 10 символов), дедлайн голосования
2. Сервер создаёт план в статусе OPEN → открывается экран деталей плана
3. Владелец добавляет участников (инвайт-ссылка / из друзей / по public ID)
4. Участники получают уведомления и присоединяются
5. Все предлагают кандидатов: даты (до 3) и места (до 5)
6. Все голосуют за предпочитаемые варианты
7. При наступлении дедлайна:
   - Если есть явный победитель → дата и место определены
   - Если ничья → включается режим «Выбор владельца» — владельцу даётся 12 часов на выбор приоритета
   - Если владелец не выбрал за 12 часов → план считается несостоявшимся и закрывается системой
8. Статус → VOTING_FINISHED
9. При наступлении даты события → статус → CLOSED
10. Участники архивируют план

**Ограничения:**
- GUEST не может создать план → показывается диалог «Требуется регистрация»
- Кандидатов дат максимум 3, мест максимум 5
- Голосовать можно только до дедлайна
- Добавлять участников может только владелец в статусе OPEN

**Развилки:**
- Ничья → режим выбора владельца (12 часов на решение)
- Владелец пропустил 12-часовой дедлайн → план считается несостоявшимся и закрывается системой
- Участник покидает план → уведомление владельцу
- Владелец удаляет план → уведомление всем участникам

### 4.3 Приглашение в план

**Три способа:**

**Способ A: Инвайт-ссылка**
1. Владелец нажимает «Добавить участника» → «По инвайт-ссылке»
2. Сервер генерирует токен и share-текст
3. Владелец делится ссылкой через любой мессенджер
4. Получатель открывает ссылку → deep link `centry://plan-invite?token=...`
5. Если приложение установлено: токен обрабатывается → пользователь добавляется в план
6. Если приложение не установлено или пользователь не авторизован: токен сохраняется локально и обрабатывается после авторизации

**Способ B: Из списка друзей**
1. Владелец → «Добавить из друзей» → выбирает друзей
2. Каждому выбранному отправляется внутреннее приглашение
3. Получатель видит модальное окно «Вас пригласили в план» с кнопками «Принять» / «Отклонить»
4. При принятии → пользователь добавляется в план
5. Владелец получает уведомление о результате

**Способ C: По публичному ID**
1. Владелец вводит public_id пользователя
2. Сервер добавляет пользователя напрямую

### 4.4 Добавление друга

**Точка входа:** кнопка «Добавить друга» на вкладке «Друзья».
**Предусловия:** нет.

**Шаги:**
1. Пользователь вводит публичный ID другого пользователя
2. Сервер проверяет статус:
   - Если уже друзья → сообщение «Уже в друзьях»
   - Если есть исходящий PENDING запрос → сообщение «Запрос уже отправлен»
   - Если есть входящий PENDING запрос → предложение «Принять сейчас?»
   - Иначе → создаётся новый PENDING запрос
3. Получатель видит модальное уведомление (если онлайн) или push
4. Получатель принимает → дружба установлена → оба получают уведомление
5. Получатель отклоняет → запрос удалён → отправитель получает уведомление

### 4.5 Отправка знака внимания

**Точка входа:** long press на карточку друга (список друзей), long press на участника плана (внутри плана или в ленте мест).
**Предусловие:** у пользователя есть неотправленный знак на сегодня.

**Шаги:**
1. Long press на друга → диалог подтверждения
2. Проверка: есть ли знак → если нет, toast «Сегодня знаков не осталось»
3. Отправка знака → знак появляется у получателя во «Входящих»
4. Получатель открывает «Коробку знаков»
5. Принимает (→ знак в коллекцию, счётчик +1) или отклоняет (→ знак исчезает)
6. Отправитель получает модальное уведомление о результате

**Особый сценарий:** если получатель принял знак от незнакомца — он может использовать «право на приглашение в друзья» (атомарно принять + отправить friend request).

### 4.6 Переписка в приватном чате

**Точка входа:** кнопка «Создать чат» у друга, или выбор существующего чата.
**Предусловие:** `canCreateChat = true` (определяется сервером).

**Шаги:**
1. Открытие чата → загрузка последних 50 сообщений
2. Написание и отправка сообщения (с дедупликацией по clientNonce)
3. Подгрузка старых сообщений при прокрутке вверх (пагинация)
4. Автоматическое обновление новых сообщений (polling каждые 2 сек)
5. Автоматическая отметка как прочитано при просмотре

**Действия с сообщениями:**
- Редактирование своего сообщения
- Удаление для всех (становится «удалённое сообщение» — tombstone)
- Удаление только для себя (скрывается)

**Действия с чатом:**
- Удалить чат
- Удалить чат и заблокировать собеседника (атомарно)

### 4.7 Удаление аккаунта

**Точка входа:** Профиль → Удалить аккаунт.

**Шаги:**
1. Экран с описанием последствий удаления
2. Чекбокс подтверждения + кнопка «Удалить навсегда»
3. Запрос на удаление отправляется на сервер (создаётся job)
4. Финализация: вызов серверной функции удаления auth-записи
5. Очистка локального хранилища
6. Выход из приложения

**Идемпотентность:** повторный запрос возвращает `already_pending`.

## 5. РОЛИ, ПРАВА И ВИДИМОСТЬ

### 5.1 Роли в плане

| Роль | Описание |
|------|---------|
| **OWNER** (Владелец) | Создатель плана. Полный контроль. |
| **PARTICIPANT** (Участник) | Приглашённый. Может голосовать, писать в чат. |

**Что может OWNER:**
- Редактировать название, описание, дедлайн
- Менять видимость плана в ленте
- Добавлять участников (только в статусе OPEN)
- Удалять участников
- Выбирать приоритет при ничьей
- Удалять весь план

**Что может PARTICIPANT:**
- Голосовать за даты и места
- Добавлять кандидатов (даты / места)
- Писать в чат плана
- Выходить из плана

**Что НЕ может PARTICIPANT:**
- Редактировать план
- Удалять других участников
- Удалять план

### 5.2 Права в контексте пользователя

**Подтверждено:** все права вычисляются на сервере и приходят клиенту как готовые булевые флаги. Клиент не принимает решений о доступности действий.

Примеры серверных флагов:
- `canEditTitle`, `canEditDescription`, `canEditDeadline` — редактирование плана
- `canDeletePlan` — удаление плана
- `canLeavePlan` — выход из плана
- `canAddMembers` — добавление участников
- `canRemoveMember` — удаление конкретного участника
- `canVote`, `canUnvote` — голосование за кандидата
- `canAddFriend` — кнопка «Добавить в друзья» у участника
- `canCreateChat` — создание чата с пользователем

### 5.3 Настройки приватности

Пользователь контролирует, что о нём видят другие в двух контекстах:

| Контекст | Описание |
|----------|---------|
| **В планах** (in_plans) | Что видят участники совместных планов |
| **В ленте** (in_feed) | Что видят пользователи в общей ленте |

Настраиваемые поля:
- Мини-профиль (краткая информация)
- Полный профиль (развёрнутая информация)

Если профиль скрыт — другие пользователи видят заглушку «Пользователь закрыл возможность просмотра».

### 5.4 Статус GUEST vs USER

| Возможность | GUEST | USER |
|------------|-------|------|
| Просмотр мест | Да | Да |
| Просмотр ленты | Да | Да |
| Создание плана | **Нет** (диалог регистрации) | Да |
| Добавление друзей | Да | Да |
| Знаки внимания | Да | Да |
| Быть приглашённым в план | Да (может голосовать, писать в чат, но не добавлять места) | Да |
| Приватные чаты | Да (если есть дружба для создания) | Да |

## 6. СОБЫТИЯ И ТРИГГЕРЫ

### 6.1 Действия пользователя → Последствия

| Действие | Что запускает |
|----------|-------------|
| Создание плана | Сервер создаёт план + чат-комнату. Бонусы (вероятно). |
| Приглашение друга в план | Уведомление получателю (PLAN_INTERNAL_INVITE). Модальное окно с кнопками Accept/Decline. |
| Принятие инвайта | Уведомление владельцу и участникам (PLAN_MEMBER_JOINED_BY_INVITE). Добавление в план. |
| Голосование за дату/место | Обновление счётчика голосов в реальном времени. |
| Наступление дедлайна | Сервер завершает голосование. Если ничья → owner choice mode. |
| Ничья без выбора владельца за 12ч | План считается несостоявшимся и закрывается системой. |
| Выход участника из плана | Уведомление владельцу (PLAN_MEMBER_LEFT). Участник исчезает из списка. |
| Удаление участника владельцем | Уведомление удалённому (PLAN_MEMBER_REMOVED). План исчезает из его списка. |
| Удаление плана | Уведомление всем non-owner участникам (PLAN_DELETED). |
| Отправка запроса в друзья | Уведомление получателю (FRIEND_REQUEST_RECEIVED). |
| Принятие запроса в друзья | Уведомление отправителю (FRIEND_REQUEST_ACCEPTED). Бонусы (вероятно). |
| Отклонение запроса | Уведомление отправителю (FRIEND_REQUEST_DECLINED). |
| Удаление из друзей | Уведомление удалённому (FRIEND_REMOVED). |
| Блокировка пользователя | Разрыв дружбы + удаление чата (атомарно). Без уведомления заблокированному. |
| Отправка знака внимания | Знак появляется у получателя во «Входящих». |
| Принятие знака | Уведомление отправителю (ATTENTION_SIGN_ACCEPTED). Знак в коллекцию. Бонусы (вероятно). |
| Отклонение знака | Уведомление отправителю (ATTENTION_SIGN_DECLINED). |
| Отправка сообщения в чат плана | Увеличение непрочитанных у всех участников. Red dot на вкладке «Мои планы». |
| Отправка приватного сообщения | Увеличение непрочитанных у получателя. Red dot на вкладке «Чаты». |

### 6.2 Серверные триггеры (по времени)

| Триггер | Когда | Что запускает |
|---------|-------|-------------|
| Дедлайн голосования | `votingDeadlineAt` наступил | Завершение голосования, определение победителей |
| Дедлайн выбора владельца | 12ч после ничьей | План закрывается как несостоявшийся |
| Дата события | `eventAt` наступил | Перевод плана в CLOSED |
| Ежедневная выдача знака | 00:00 UTC | Новый знак внимания для каждого пользователя |
| Истечение знака | 00:00 UTC следующего дня | Неотправленный знак пропадает |
| Напоминание о голосовании | Перед дедлайном | Push: PLAN_VOTING_REMINDER_DATE/PLACE/BOTH |
| Напоминание владельцу о ничьей | < 12ч до дедлайна | Push: PLAN_OWNER_PRIORITY_DATE/PLACE/BOTH |
| Напоминание о событии | 24ч до eventAt | Push: PLAN_EVENT_REMINDER_24H |

### 6.3 Модальные окна (когда появляются)

Модальные окна показываются при:
- Открытии приложения (проверка очереди)
- Возобновлении из background
- Получении realtime-уведомления (подписка на INBOX)
- Получении push-уведомления в foreground

Модальные окна показываются по одному, в порядке очередности (от старого к новому). После показа каждое отмечается как обработанное (consumed).

## 7. СТАТУСЫ И ЖИЗНЕННЫЕ ЦИКЛЫ

### 7.1 Пользователь

```
[Первый вход] → GUEST → [Подтверждение email] → USER
                                                    ↓
                                         [Удаление аккаунта] → Удалён
```

| Статус | Что означает | Доступные действия |
|--------|-------------|-------------------|
| GUEST | Ник создан, email не подтверждён | Просмотр мест/ленты, друзья, знаки. Нельзя создавать планы. |
| USER | Полноценный пользователь | Все функции |

### 7.2 План

```
        ┌──────────┐
        │   OPEN   │ ← Голосование идёт
        └────┬─────┘
             │ (дедлайн голосования наступил)
             ↓
   ┌─────────────────────┐
   │  VOTING_FINISHED    │ ← Результаты определены
   │ (или owner choice)  │   (или ничья → выбор владельца)
   └─────────┬───────────┘
             │ (дата события наступила)
             ↓
        ┌──────────┐
        │  CLOSED  │ ← Архивный
        └──────────┘
```

| Статус | Что доступно | Что недоступно |
|--------|-------------|---------------|
| **OPEN** | Голосование, добавление кандидатов, добавление участников, чат, редактирование | — |
| **VOTING_FINISHED** | Просмотр результатов, чат, выход, архивация | Голосование, добавление кандидатов, добавление участников |
| **CLOSED** | Просмотр, архивация | Всё остальное |

**Флаг archived** — ортогонален статусу. Пользователь может вручную архивировать план на любом этапе. Архивные планы отображаются на отдельной вкладке.

### 7.3 Запрос в друзья

```
[Отправка запроса] → PENDING
                       ├─→ [Принятие] → Дружба установлена
                       └─→ [Отклонение] → Запрос удалён
```

### 7.4 Дружба

```
[Дружба установлена]
    ├─→ [Удаление из друзей] → Дружба разорвана
    └─→ [Блокировка] → Дружба разорвана + чат удалён + блокировка создана
```

### 7.5 Знак внимания

```
[00:00 UTC: выдача знака]
    ├─→ [Отправка другу]
    │      ↓
    │   [Во входящих у получателя]
    │      ├─→ [Принятие] → В коллекцию (count +1)
    │      └─→ [Отклонение] → Знак исчезает
    └─→ [Не отправлен до 00:00] → Знак пропадает
```

### 7.6 Приватный чат

```
[Создание чата]
    ↓
[Активный чат] ← обмен сообщениями
    ├─→ [Удаление чата] → Чат удалён
    └─→ [Удаление + блокировка] → Чат удалён + пользователь заблокирован
```

### 7.7 Сообщение в чате

```
[Отправка] → Активное сообщение
                ├─→ [Редактирование] → Отредактированное (edited_at)
                ├─→ [Удаление для всех] → Tombstone (видно как "Сообщение удалено")
                └─→ [Удаление для себя] → Скрыто только для автора
```

## 8. МАТРИЦА ПОВЕДЕНИЯ ПО ЭКРАНАМ И БЛОКАМ

### 8.1 Экран «Места»

| Элемент | Содержимое | Зависит от |
|---------|-----------|-----------|
| Список мест | Карточки мест с фото, названием, категорией, рейтингом, расстоянием, метро | Фильтры (город, район, категория, рейтинг), геопозиция |
| Карта | OpenStreetMap с маркерами, кластеризация при зуме < 16, лейблы при зуме ≥ 15.8 | Viewport карты, геопозиция |
| Фильтры | Город, район, категория (мультиселект), мин. рейтинг | Сохраняются между сессиями |
| Поиск | Автодополнение по названию | Ввод пользователя |

**Пустые состояния:**
- Нет мест по фильтрам → пустой список
- Нет геопозиции → сервер отдаёт случайную выборку / сортировка по умолчанию

**CTA:** Карточка места → Детали места (модал). Кнопка «Добавить место» → форма подачи.

### 8.2 Экран «Мои планы»

| Элемент | Содержимое | Зависит от |
|---------|-----------|-----------|
| Вкладка «Активные» | Карточки планов (название, статус, участники, дедлайн) | Список планов пользователя |

### 9.2 Каналы доставки

1. **Realtime (основной)** — подписка на серверную таблицу доставок. Мгновенная реакция если приложение открыто.
2. **Push (FCM)** — Firebase Cloud Messaging. Для событий когда приложение в background. Работает через серверную edge function `push_worker`.
3. **Модальная очередь** — при открытии/возобновлении приложения проверяется очередь непоказанных событий.

### 9.3 Grace Period (защита от дубликатов в push)

Серверный push_worker проверяет, было ли событие уже обработано в приложении через realtime (INBOX.CONSUMED), и использует «grace period»:
- 15 секунд для большинства событий
- 60 секунд для чат-сообщений

Если событие было обработано в приложении до истечения grace period — push не отправляется.

### 9.4 Дедупликация на клиенте

Каждый тип координатора ведёт свой набор обработанных ID:
- `InviteUiCoordinator` — по inviteId
- `PlanMemberLeftUiCoordinator` — по (planId, userId) с TTL 5 сек
- `PlanDeletedUiCoordinator` — по (planId, ownerUserId) с TTL 5 сек
- `PlanScheduledNotificationUiCoordinator` — bounded set до 300 ключей

Это предотвращает показ одного и того же события дважды (например, через realtime + push).

## 10. ПОГРАНИЧНЫЕ И НЕОЧЕВИДНЫЕ СЦЕНАРИИ

### 10.1 Незавершённые действия

| Сценарий | Что происходит |
|----------|---------------|
| Пользователь начал онбординг, но не завершил email | Остаётся GUEST. При следующем открытии — сразу главный экран (без email). |
| Пользователь получил инвайт-ссылку, но не авторизован | Токен сохраняется локально. После авторизации — автоматически обрабатывается. |
| Пользователь закрыл приложение во время голосования | Голос уже отправлен на сервер при нажатии. Если не нажал — голос не засчитан. |
| Владелец не выбрал приоритет при ничьей за 12 часов | План закрывается системой как несостоявшийся. |

### 10.2 Изменённые данные

| Сценарий | Что происходит |
|----------|---------------|
| План удалён пока пользователь смотрит его | Realtime-событие скрывает план из списка. При следующем обновлении — ошибка «План больше недоступен». |
| Друг заблокировал пользователя | Дружба разрывается. Чат удаляется. Изменения отражаются при следующем обновлении списков. |
| Версии юридических документов обновились | При следующем открытии — показ экрана переакцептации (блокирующий). |
| Участник удалён из плана пока он его просматривает | Модальное уведомление «Вас удалили из плана». План исчезает из списка. |

### 10.3 Удалённые / отклонённые / недоступные объекты

| Сценарий | Что происходит |
|----------|---------------|
| Переход по инвайт-ссылке в удалённый план | Ошибка от сервера: «План больше недоступен или у вас нет доступа». |
| Попытка написать в чат удалённого плана | Ошибка при загрузке деталей. |
| Пользовательское место на модерации | Показывается с пометкой «На модерации» (isPendingModeration). Нельзя голосовать. |
| Пользовательское место отклонено | Показывается с пометкой «Отклонено» (isRejected). |

### 10.4 Истёкшие сроки

| Сценарий | Что происходит |
|----------|---------------|
| Дедлайн голосования прошёл | Голосование автоматически закрывается. Статус → VOTING_FINISHED. |
| Знак внимания не отправлен до 00:00 | Знак пропадает. Новый выдаётся на следующий день. |
| Срок знака во входящих истёк | Знак исчезает из входящих (expiresAt). |
| Модальное событие истекло | При consume сервер возвращает `skip: true` → событие не показывается. |

### 10.5 Изменение роли / статуса

| Сценарий | Что происходит |
|----------|---------------|
| GUEST пытается создать план | Диалог «Требуется регистрация» с кнопкой «Зарегистрироваться» → экран email. |
| Пользователь подтвердил email (стал USER) | Auth state change → полная переинициализация → доступны все функции. |

### 10.6 Дубликаты и повторные действия

| Сценарий | Что происходит |
|----------|---------------|
| Повторная отправка запроса в друзья | Сервер возвращает существующий PENDING-запрос. |
| Повторный запрос на удаление аккаунта | Сервер возвращает `already_pending`. |
| Повторный tap по push-уведомлению | Дедупликация на клиенте по ключу события. |
| Повторная отправка сообщения | Дедупликация по clientNonce. |

### 10.7 Возврат в середину процесса

| Сценарий | Что происходит |
|----------|---------------|
| App возобновляется из background | Обновление геопозиции, проверка FCM-токена, heartbeat, проверка модальной очереди. |
| Пользователь вернулся к экрану плана | Автообновление каждые 10 сек. При resume — немедленная загрузка + повторная через 700ms (eventual consistency). |
| Пользователь вернулся к списку чатов | Немедленная перезагрузка + polling каждые 5 сек. |

### 10.8 Геолокация отсутствует

| Сценарий | Что происходит |
|----------|---------------|
| Нет разрешения на геолокацию | Сервер отдаёт места без сортировки по расстоянию (режим DEFAULT вместо GEO). |
| GPS недоступен | Используется последняя известная позиция (сохранена в защищённом хранилище). |
| Нет ни текущей, ни сохранённой позиции | Лента показывает случайную выборку мест. |

## 11. ЧТО В ПРОДУКТЕ ВЫГЛЯДИТ АКТУАЛЬНЫМ, А ЧТО СПОРНЫМ

### 11.1 Подтверждённое актуальное поведение

- **Онбординг** — полный рабочий цикл с GUEST → USER, юридическими документами, OTP
- **Планы** — полный жизненный цикл: создание → голосование → ничья/выбор → событие → архив
- **Чат в планах** — рабочий, с пагинацией, edit/delete, бейджами непрочитанных
- **Приватные чаты** — полноценная переписка с edit/delete/bulk delete
- **Друзья** — запросы, принятие/отклонение, заметки, удаление
- **Знаки внимания** — ежедневная механика, коробка с тремя секциями, коллекция
- **Блокировка** — атомарное действие (разрыв дружбы + удаление чата)
- **Бонусы** — баланс, история, правила начисления
- **Лидерборд** — два режима (активность + симпатии)
- **Уведомления** — realtime + push + модальная очередь + дедупликация
- **Места** — каталог с фильтрами, картой, голосованием, подачей новых мест
- **Юридические документы** — версионирование с принудительной переакцептацией

### 11.2 Спорные / неочевидные места

| Область | Наблюдение | Требует проверки |
|---------|-----------|-----------------|
| **Разблокировка** | Разблокировка не реализована ни на клиенте, ни на сервере. Это сознательное решение — функция планируется к релизу. | Подтверждено. |
| **GUEST и планы** | GUEST не может создавать планы, но **может быть приглашён** в план. Guest может голосовать и писать в чат, но не может добавлять места-кандидаты. | Подтверждено. |
| **Feed/Лента** | Сигналы (interested/planned/visited) — автоматические агрегаты из активности вокруг мест через планы. Пользователь их не ставит вручную. Interested = место в голосовании, planned = место победило, visited = событие прошло. | Подтверждено. |
| **Centry Market** | Есть только накопление токенов. Трата пока не реализована. Market с возможностью траты откроется с релизом. Все накопленные токены сохранятся. | Подтверждено. |
| **Submission мест** | Пользователь может подать новое место, но механизм модерации виден только как статусы (pending/rejected). Нет экранов модерации в клиенте. | Модерация происходит через внешний инструмент? Или автоматически? |
| **Таблица core_activities** | На сервере существует таблица `core_activities` (27 колонок, 0 записей). Возможно, это будущая фича для событий/активностей при местах. | Что планируется для activities? |
| **Identity guests/users** | Таблицы `identity_users` и `identity_guests` существуют, но пусты (0 записей). При этом `app_users` содержит 137 записей. | Является ли identity-система устаревшей или будущей? |
| **Polling вместо WebSocket для чатов** | Приватные чаты обновляются polling (каждые 2–5 сек), а не через realtime-подписку. Это создаёт задержку до 5 сек для новых сообщений. | Планируется ли переход на realtime для чатов? |
| **Online-статус** | Heartbeat отправляется при открытии и resume приложения, но механизм показа «онлайн/оффлайн» другим пользователям неявен в клиентском коде. | Как отображается онлайн-статус и где он виден? |
| **Масштаб данных мест** | 9400+ мест в базе, 16600+ результатов поиска. При этом staging-таблицы содержат тысячи записей. Процесс импорта и обогащения мест сложный и многоступенчатый. | Как устроен pipeline добавления новых мест? Автоматический или ручной? |

### 11.3 Признаки старой / наследованной логики

| Признак | Описание |
|---------|---------|
| `identity_users` / `identity_guests` | Пустые таблицы, возможно, от предыдущей модели идентификации |
| Множество staging-таблиц мест | `staging_places_raw`, `staging_areas_polygons`, `staging_metro_raw` — вероятно, от процессов первичного импорта данных |

| `search_batch_test` | Тестовая таблица рядом с рабочей `search_batch` |

## 12. ИТОГОВАЯ БАЗА ЗНАНИЙ В КОМПАКТНОМ ВИДЕ

### Продукт в одном абзаце

**Centry** — мобильное приложение для совместного планирования досуга с друзьями. Пользователи находят места (рестораны, бары, клубы и др.) в каталоге с картой и фильтрами, создают планы-встречи с голосованием за дату и место, приглашают друзей через ссылки или из списка контактов, общаются в чатах планов и личных переписках, обмениваются ежедневными знаками внимания и зарабатывают бонусные токены за активность. Вся бизнес-логика вычисляется на сервере (Supabase/Postgres); клиент только рендерит готовые данные.

### Ключевые сущности

1. **Пользователь** (GUEST → USER) — профиль, аватар, предпочтения, приватность
2. **Место** (9400+ в каталоге) — заведение с категорией, рейтингом, координатами
3. **План** (OPEN → VOTING_FINISHED → CLOSED) — встреча с голосованием, чатом, участниками
4. **Дружба** (PENDING → принята / отклонена) — двусторонняя связь
5. **Приватный чат** — переписка между двумя пользователями
6. **Знак внимания** — ежедневный символ симпатии с коллекцией
7. **Бонусные токены** — виртуальная валюта за активность
8. **Блокировка** — защита от нежелательных контактов

### Ключевые сценарии

1. Онбординг: ник → разрешения → email → главный экран
2. Создание плана → приглашение друзей → голосование → встреча
3. Добавление друга по public ID → принятие запроса
4. Поиск места → фильтры/карта → сохранение / добавление в план
5. Личная переписка между друзьями
6. Ежедневный знак внимания → отправка → принятие/отклонение
7. Заработок и просмотр бонусных токенов
8. Блокировка нежелательного пользователя

### Основные статусы и роли

- **Пользователь:** GUEST / USER
- **План:** OPEN / VOTING_FINISHED / CLOSED (+ archived)
- **Роль в плане:** OWNER / PARTICIPANT
- **Запрос в друзья:** PENDING (OUTGOING / INCOMING)
- **Submission места:** pending / одобрено / rejected

### Главные триггеры

- Дедлайн голосования → автозавершение
- Ничья → owner choice mode (12ч) → план закрывается если не выбрал
- 00:00 UTC → новый знак внимания
- 24ч до события → напоминание
- Блокировка → разрыв дружбы + удаление чата
- Обновление юридических документов → принудительная переакцептация

### Самые важные ограничения

- GUEST не может создавать планы
- Максимум 3 кандидата дат и 5 кандидатов мест в плане
- Один знак внимания в день, пропадает если не отправлен
- Голосование закрывается строго по дедлайну
- Разблокировка не реализована (планируется к релизу)
- Все права и доступы вычисляются на сервере
