# Plan.md — Дорожная карта: нормализация базы мест Centry

> Дата составления: 2026-04-03
> Объём: ~9000 мест, 8 городов, более 9000 фото парсинга

---

## Контекст

Текущее состояние:
- **core_places** — ядро мест (~9000 записей, 8 городов)
- **place_enrichment** — таблица обогащения (фото, рейтинг, метро, сайт, телефон)
- **place_search_results_raw** — сырые данные парсинга Яндекс.Карт (provider=yandex_maps)
- **scraped_photos/** — более 10000 webp файлов (до 3 фото на место, 600×320, quality 82)
- **brand-media** — Supabase Storage bucket для фото мест

Клиент использует:
- `get_places_feed_v5` (лента) и `get_feed_nearby` (activity feed)
- `PlaceDto` / `FeedPlaceDto` — маппят поля из enrichment (preview_storage_key, rating, metro, website)
- Плейсхолдеры по категориям: bar(32), restaurant(37), nightclub(19), cinema(15), theatre(20), hookah(1), karaoke(1), banya_sauna(1)

## Общая декомпозиция

| # | Сессия | Суть | Входные данные | Выход |
|---|--------|------|----------------|-------|
| 1 | **TZ1** | Аудит и фильтрация — убираем нерелевантные места | core_places + place_search_results_raw | staging_places_clean (отфильтрованная таблица) |
| 2 | **TZ2** | Нормализация данных — адреса, телефоны, сайты, выбор фото | staging_places_clean + scraped_photos | staging_places_normalized |
| 3 | **TZ3** | Заливка в прод — обновление enrichment, фото на Storage, рейтинг, проверка клиента | staging_places_normalized | обновлённая place_enrichment + brand-media |

---

## Почему 3 сессии, а не 1

1. **Объём данных**: ~9000 мест × (фильтрация + нормализация адресов + валидация телефонов + отбор фото + конвертация + заливка + рейтинг) — каждый шаг требует проверки и контроля качества.

2. **Точки контроля**: после каждой сессии — промежуточный результат, который можно проверить. Если на шаге 1 фильтрация слишком агрессивная или мягкая — корректируем до заливки в прод.

3. **Контекст сессии**: обработка 9000 мест с SQL-запросами, скриптами обработки фото и валидацией быстро перегрузит контекст. Разбивка даёт чистый старт каждой сессии.

4. **Безопасность**: данные парсинга сохраняются как черновик (staging), прод обновляется только в финальной сессии после всех проверок.

## Зависимости и порядок

```
TZ1 (аудит + фильтрация)
  ↓
TZ2 (нормализация данных)
  ↓
TZ3 (заливка в прод + рейтинг + проверка клиента)
```

Каждая сессия зависит от результатов предыдущей. Параллельно выполнять нельзя.

## Критерии готовности (Definition of Done)

После выполнения всех трёх сессий:

- [ ] Все нерелевантные места удалены из core_places (или деактивированы)
- [ ] Каждое оставшееся место имеет:
  - Релевантное название и тип (restaurant, bar, nightclub, cinema, theatre, hookah, karaoke, bathhouse)
  - Корректный адрес (до корпуса/строения максимум, без подъездов/этажей)
  - Валидные координаты
  - Привязку к району и городу
  - Одно качественное фото (или плейсхолдер по типу)
  - Нормализованный телефон (или пусто)
  - Валидный сайт (или пусто) — без ссылок на 2ГИС/Яндекс/VK/Telegram
  - Рейтинг с реалистичным распределением лайков/дизлайков (±0.1 от Яндекса)
  - Метро + расстояние (где применимо)
- [ ] Старые ненужные фото удалены из brand-media Storage
- [ ] Данные парсинга сохранены в staging-таблице как черновик
- [ ] Клиент корректно отображает карточки мест (лента, карта, детали, план)
- [ ] Нет регрессий: фильтры, поиск, голосование в планах, сохранение мест работают

## Риски

| Риск | Митигация |
|------|-----------|
| Парсинг не покрывает все места из core_places | Места без данных парсинга — оставляем текущие данные или ставим плейсхолдер |
| Ложные срабатывания фильтрации (удалили нужное) | Staging-таблица, всё проверяем до применения к проду |
| Фото с лицами людей прошли фильтр | Ручная выборочная проверка после автоматической |
| Рейтинг не сходится с Яндексом | Допуск ±0.1, формула пересчёта через лайки/дизлайки |
| Удаление мест ломает существующие планы | Проверяем привязки к plan_candidates перед удалением |
