# Прокачка NotebookLM Knowledge Base — bootstrap чеклист

## Цель

Залить в NotebookLM `Centry — Knowledge Base` baseline-знания о проекте, чтобы любая будущая сессия могла через `notebooklm ask` поднять исторический и архитектурный контекст без необходимости каждый раз исследовать всё с нуля.

## Параметры

- **Notebook ID:** `83e2a76b-2e04-4280-885e-32fb0b005a19`
- **Notebook title:** `Centry — Knowledge Base`
- **CLI:** `notebooklm` (нужен `export PATH="$HOME/bin:$PATH"`)
- **MCP:** Supabase MCP подключён, есть `mcp__supabase__list_tables`, `execute_sql`, `list_edge_functions` и т.д.

## Жёсткие правила (CLAUDE.md правило 4)

1. **Только реальные факты.** Каждое утверждение в source — результат реального запроса к Supabase / реального чтения файла. Никаких гипотез, никакого «вероятно работает так».
2. **Если не знаешь как устроено — так и пиши** в source: «не исследовано» или «требует уточнения». Не угадывай.
3. **Не дублируй то, что есть в коде/git.** Source должен содержать **связи, концепции и почему**, а не дамп файла. Дамп схемы БД — это исключение, потому что схему быстро не вытащишь.
4. **Один блок = одна сессия = один source.** Не пытайся за раз сделать несколько блоков — контекст переполнится и качество упадёт.
5. **Перед заливкой — покажи юзеру черновик** или хотя бы оглавление, чтобы он мог поправить акценты до записи.

## Алгоритм одной сессии

1. Открой этот файл, найди **первый блок без галочки `[x]`**.
2. Прочитай его секцию «Что исследовать» и «Что попадает в source».
3. Выполни исследование строго по описанному алгоритму. Не выходи за рамки блока.
4. Сформируй markdown-файл во временной директории `/tmp/centry_kb_block<N>_<slug>.md` со структурой из секции «Шаблон source».
5. **Покажи пользователю краткое оглавление того, что собрал**, и спроси: «заливать или поправить?»
6. После подтверждения — залей через `notebooklm source add /tmp/... --notebook 83e2a76b-2e04-4280-885e-32fb0b005a19 --title "<заголовок из блока>"`.
7. Удали временный файл.
8. **Отметь галочку** `[x]` в этом файле для соответствующего блока. Допиши под блоком короткую запись «выполнено YYYY-MM-DD, source ID `<id>`».
9. Сообщи пользователю, что блок закрыт, какой следующий, и предложи завершить сессию.

### [x] Блок 2 — RPC и edge functions

> выполнено 2026-04-09, source ID `6a281823-8d85-4fc9-b4be-8c5b0aac5a60`

**Заголовок source:** `KB Block 2 — RPC functions and edge functions`

**Что исследовать:**
- `mcp__supabase__list_edge_functions` — все edge functions, их пути
- Через `execute_sql`: `SELECT proname, pronargs, prorettype::regtype FROM pg_proc WHERE pronamespace = 'public'::regnamespace ORDER BY proname` — список всех RPC в public
- Для функций с понятными именами (`get_*_snapshot`, `*_for_app_user`) — кратко, что возвращают, по сигнатуре
- НЕ читать тела функций — это огромный объём, в NotebookLM запросим через ask при необходимости

**Что попадает в source:**
- Список edge functions
- Список RPC в public, сгруппированный по доменам (на основе префиксов имён)
- Выделить **server-first snapshot RPC** (функции с `_snapshot` в имени или возвращающие jsonb агрегаты) — это критичный паттерн проекта
- Выделить функции, принимающие `app_user_id` параметр

**Чего НЕ делать:**
- НЕ описывать что функция делает «по догадке» — только то, что явно из сигнатуры/имени
