# TZ1 — Аудит базы и фильтрация нерелевантных мест

> Сессия 1 из 3
> Зависимости: скрипт парсинга завершён, данные в place_search_results_raw

---

## Цель сессии

Исследовать текущую базу мест, сверить с данными парсинга Яндекса, убрать нерелевантные записи. На выходе — чистый список мест, пригодных для продукта. Места 
в продукте должны соответствовать тематике приложения - Места для встречь с друзьями: Бары, Ночные клубы, Рестораны, Караоке, Кальянные,Театры, Кинотеатры, Бани и Сауны. Все что не соответствует должно быть удалено из базы. Не должно быть ни каких ларьков, фастфудов, магазинов, автосервисов и так далее.
---

## Шаг 1: Исследование текущей базы

### 1.1 Структура и объёмы
- Получить полную структуру таблиц: `core_places`, `place_enrichment`, `place_search_results_raw`
- Посчитать общее количество мест по городам и категориям
- Посчитать сколько мест имеют данные в place_search_results_raw (покрытие парсинга)
- Определить сколько мест без парсинга вообще

### 1.2 Связи с другими сущностями
- Проверить связи core_places с: `plan_candidates`, `saved_places`, `place_votes`, `place_enrichment`
- Составить карту зависимостей — какие места нельзя просто удалить (используются в активных планах)

### 1.3 Текущие категории
- Выгрузить все уникальные значения category из core_places
- Сопоставить с допустимыми типами: `bar`, `restaurant`, `nightclub`, `cinema`, `theatre`, `hookah`, `karaoke`, `bathhouse`
- Выявить "мусорные" категории

## Шаг 2: Определение критериев релевантности

### 2.1 Портрет допустимого места
Место подходит для Centry если это заведение где люди могут **встречаться и проводить время вместе**:
- Рестораны, кафе, бары, пабы, гастробары
- Ночные клубы, лаунж-бары
- Кинотеатры, театры
- Кальянные, караоке
- Бани, сауны
- Антикафе, коворкинги (с социальным форматом)
- Боулинг, бильярд, развлекательные центры

### 2.2 Что точно НЕ подходит (стоп-лист)
- Компьютерные клубы, киберспорт-арены
- Мелкие пивные магазины, продуктовые точки, ларьки
- Забегаловки без нормального интерьера
- Гостиницы, хостелы, отели
- Автосервисы, шиномонтаж
- Институты, школы, университеты
- Магазины расходников, запчастей, электроники
- Батарейки, вода, комплектующие (случайный мусор)
- Аптеки, поликлиники, мед.центры
- Офисы, бизнес-центры
- Салоны красоты, барбершопы (не социальный формат)
- Ритуальные услуги
- Любые точки без понятного "социального" назначения

## Шаг 3: Создание staging-таблицы

### 3.1 Создать `staging_places_audit`
```sql
CREATE TABLE staging_places_audit (
  place_id UUID PRIMARY KEY REFERENCES core_places(id),
  city_name TEXT,
  category TEXT,
  title TEXT,
  yandex_title TEXT,          -- из парсинга
  yandex_category TEXT,       -- категория по Яндексу
  yandex_rating NUMERIC,      -- рейтинг Яндекса
  has_photos BOOLEAN,         -- есть ли фото в парсинге
  photo_count INT,            -- сколько фото скачано
  verdict TEXT NOT NULL,       -- 'keep' | 'remove' | 'review'
  reason TEXT,                 -- почему remove/review
  in_active_plan BOOLEAN DEFAULT FALSE,  -- используется ли в активном плане
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 3.2 Наполнить таблицу
- JOIN core_places + place_search_results_raw
- Автоматически проставить verdict='remove' для очевидного мусора (по названию, категории Яндекса, стоп-листу)
- verdict='keep' для мест с релевантными категориями и хорошим рейтингом
- verdict='review' для пограничных случаев

## Шаг 4: Автоматическая фильтрация

### 4.1 По категориям (category из core_places)
- Все места с допустимыми категориями → verdict='keep' (предварительно)
- Все места с неизвестными/мусорными категориями → verdict='remove'

### 4.2 По названиям (yandex_title и title)
- Regex-фильтры по стоп-словам в названии:
  - `компьютерн|кибер|gaming|game zone`
  - `пивн.*магаз|разлив|на.*розлив`
  - `гостиниц|отель|хостел|hotel|hostel`
  - `автосервис|шиномонтаж|автомойка`
  - `институт|университет|школа|колледж|академия`
  - `аптека|поликлиник|больниц|мед.*центр`
  - `ритуальн|похорон`
  - `магазин|маркет|торгов|оптов` (кроме "гастромаркет", "фудмаркет")
  - `офис|бизнес.центр`
  - `салон.*красот|барбер|парикмахер`

### 4.3 По данным парсинга
- Места где парсинг дал `no_results` → verdict='review' (возможно место не существует)
- Места без данных парсинга вообще → verdict='review'

### 4.4 Защита от удаления
- Места в активных планах (OPEN/VOTING_FINISHED) → in_active_plan=TRUE, verdict не может быть 'remove'
- Места с сохранениями пользователей → пометить, но можно удалять

## Шаг 5: Статистика и отчёт

### 5.1 Выгрузить сводку
- Сколько keep / remove / review по городам
- Сколько мест с активными планами попали в remove → перевести в review
- Топ причин удаления
- Покрытие парсингом (% мест с данными Яндекса)

### 5.2 Применение фильтрации
После ревью статистики:
- Места с verdict='remove' → soft-delete или деактивация в core_places (НЕ физическое удаление)
- Механизм деактивации: обсудить — добавить поле `is_active` или переместить в отдельную таблицу

## Выходные артефакты

1. **staging_places_audit** — таблица с вердиктами по каждому месту
2. **Статистика** — количество мест до/после по городам и категориям
3. **Список review** — пограничные случаи для ручной проверки (если есть)
4. **core_places** — деактивированные нерелевантные записи
