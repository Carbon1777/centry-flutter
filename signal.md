# Карта сигналов Centry: пуши, модалки, бейджи

## Архитектура доставки

Каждое событие в системе проходит через серверную очередь и доставляется по **двум каналам параллельно**:

- **INBOX** — realtime-подписка (Supabase Realtime). Работает мгновенно, пока приложение открыто.
- **PUSH** — Firebase Cloud Messaging. Работает, когда приложение в фоне или убито. Имеет **grace period** (задержку 15–60 сек), чтобы дать INBOX шанс доставить первым.

Если INBOX успел доставить и показать модалку — PUSH-доставка помечается как CONSUMED и не отправляется.

**Модальные события** — отдельная очередь (`modal_event_queue`). Клиент при открытии/resume запрашивает все PENDING модалки и показывает их последовательно (FIFO, от старых к новым).

**Бейджи** — считаются на сервере, клиент получает готовые числа через RPC polling (каждые 5-10 сек).

## Сценарии по доменам

### 1. Друзья

#### 1.1 Запрос в друзья (отправлен)

| Получатель | Что происходит |
|------------|---------------|
| **Invitee (кому отправлен)** | **Модалка**: диалог с аватаром «@nick хочет добавить тебя в друзья» + кнопки «Принять» / «Отклонить» |
| | **Push**: data-only (нет OS-уведомления, только триггерит модалку при открытии) |
| | **Бейдж**: нет |

#### 1.2 Запрос в друзья — принят

| Получатель | Что происходит |
|------------|---------------|
| **Requester (кто отправлял)** | **Модалка**: информационный диалог «@nick принял твой запрос в друзья» |
| | **Push**: data-only |
| | **Бейдж**: нет |

#### 1.3 Запрос в друзья — отклонён

| Получатель | Что происходит |
|------------|---------------|
| **Requester** | **Модалка**: информационный диалог «@nick отклонил твой запрос в друзья» |
| | **Push**: data-only |
| | **Бейдж**: нет |

#### 1.4 Удалён из друзей

| Получатель | Что происходит |
|------------|---------------|
| **Удалённый пользователь** | **Модалка**: информационный диалог «@nick удалил тебя из друзей» |
| | **Push**: data-only |
| | **Бейдж**: нет |

### 6. Чат плана

#### 6.1 Новое сообщение в чате плана

| Получатели | Что происходит |
|------------|---------------|
| **Все участники (кроме автора)** | **Модалка**: нет |
| | **Push**: data-only (grace period 60 сек, батчинг — несколько сообщений объединяются в одну доставку) |
| | **Бейдж**: да — красная точка на карточке плана в списке планов + счётчик непрочитанных в чате + красная точка на вкладке «Мои планы» в нижней навигации |

**Поведение при открытом приложении:**
- Push НЕ показывается (foreground listener подавляет)
- Бейджи обновляются через polling каждые 10 сек
- При открытии чата — сообщения приходят через snapshot polling каждые 2 сек

**Поведение при тапе на push:**
- Открывает приложение (без навигации к конкретному плану)

**Поведение при тапе на push:**
- Открывает экран списка приватных чатов (PrivateChatsListScreen)

## Механики доставки в деталях

### Grace period (задержка перед отправкой push)

Push не отправляется мгновенно — сервер ждёт, чтобы дать INBOX (realtime) возможность доставить событие первым:

| Тип события | Grace period |
|-------------|-------------|
| Все PLAN_* (кроме чата) | 15 сек |
| Все FRIEND_* | 15 сек |
| PLAN_CHAT_MESSAGE | 60 сек |
| PRIVATE_CHAT_MESSAGE | 60 сек |

### Батчинг сообщений чата

При множестве сообщений в чате — push НЕ создаётся на каждое сообщение. Вместо этого:
1. Первое сообщение — создаётся PENDING push-доставка
2. Следующие сообщения — обновляется timestamp существующей PENDING доставки
3. Push отправляется только когда grace period истекает без новых сообщений

### Дедупликация модалок

Клиент хранит set из уже показанных event_id (до 300 штук) и не показывает одно и то же событие дважды.

### Триггеры проверки модалок на клиенте

`checkAndShowModalEvents()` вызывается:
1. При запуске приложения (после загрузки профиля)
2. При resume из фона (app lifecycle)
3. При получении INBOX realtime-события
4. При получении FCM push в foreground (для не-чатовых типов)
5. После обработки тапа по push-уведомлению

### Обновление бейджей

Бейджи обновляются через polling:
- `get_my_plan_chat_badges_v1` — каждые 10 сек на экране планов
- `get_private_chat_badges_v1` — каждые 5 сек на экране списка чатов
- Дополнительно при resume из фона

### Отметка как прочитано

- `mark_plan_chat_read_v1(p_read_through_room_seq)` — при открытии чата плана
- `mark_private_chat_read_v1(p_read_through_seq)` — при открытии приватного чата
- Автоматически при каждом получении snapshot (если чат открыт)
