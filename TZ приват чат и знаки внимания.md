ТЗ: Приватные чаты, знаки внимания, блокировки
1. Роль и формат работы

Нужен архитектор/кодер, который работает строго в модели server-first.

Ключевой принцип проекта:

сервер — источник истины;
клиент — только рендер и навигация;
критическая бизнес-логика не должна расползаться по клиенту.

Перед реализацией обязательно:

исследовать текущую реализацию внутрипланового чата на сервере и клиенте;
исследовать уже существующую реализацию дружбы / приглашений в друзья;
определить, что можно безопасно переиспользовать;
затем реализовать новый блок без поломки текущих сценариев.
2. Цель задачи

Реализовать новый функциональный блок:

Приватные чаты 1-на-1
Список приватных чатов как 4-й раздел ленты
Создание приватного чата только с другом
Удаление чата
Удаление чата с блокировкой пользователя
Экран блокировок в профиле
Система знаков внимания
Переход от принятого знака внимания к возможности отправить приглашение в друзья
Полная совместимость с server-first архитектурой
3. Что уже существует и что нужно исследовать
3.1. Внутриплановый чат

Архитектор обязан сначала исследовать уже реализованный чат внутри планов:

серверную модель;
хранение сообщений;
realtime;
статусы сообщений;
непрочитанные;
поведение при foreground/background/cold start;
уведомления;
клиентский рендер;
шапку чата;
навигацию;
жизненный цикл экрана;
поведение клавиатуры.

Новый приватный чат по механике должен быть максимально близок к внутриплановому чату, если ниже не зафиксировано иное поведение.

3.2. Дружба / приглашения в друзья

Нужно исследовать уже существующий механизм:

отправка приглашения в друзья по ID;
отправка приглашения в друзья из списка участников плана;
принятие / отклонение;
текущее хранение дружбы;
текущие уведомления и модалки.

Новый механизм дружбы придумывать не нужно. Нужно использовать уже существующий.

4. Общий функциональный контур

Новый блок состоит из трёх связанных систем:

4.1. Приватные чаты
отдельный раздел приложения;
список всех приватных чатов пользователя;
переход в конкретный чат;
чат строго 1-на-1;
чат можно создать только с другом;
один активный чат на одну пару пользователей.
4.2. Знаки внимания
пользователь получает один случайный знак внимания в день;
знак можно отправить только из списка участников видимого плана в ленте, из списка друзей и из списка участников планан в самом плане;
получатель может принять или отклонить знак;
после принятия отправителю даётся разовая возможность отправить приглашение в друзья.
4.3. Блокировки
пользователь может заблокировать другого:
из приватного чата;
из списка друзей;
блокировка разрывает дружбу;
удаляет приватный чат;
удаляет историю;
запрещает дальнейшие взаимодействия.
5. Архитектурные принципы реализации
5.1. Server-first

Все проверки серверные:

можно ли создать чат;
есть ли уже чат;
есть ли дружба;
есть ли блокировка;
можно ли отправить знак;
истёк ли знак;
можно ли принять/отклонить;
можно ли показать кнопку;
есть ли право отправить приглашение в друзья после принятия знака;
можно ли писать в чат;
можно ли удалить чат;
что показывать в списках и индикаторах;
какие модалки показывать при входе.
5.2. Клиент без локальной бизнес-логики

Клиент:

вызывает серверное действие;
получает канонический результат;
рендерит состояние;
показывает тост/модалку/экран по контракту.
5.3. Чатовая механика

Сообщения, статусы, realtime и базовое UX-поведение брать из уже существующего внутрипланового чата:

либо путём прямого переиспользования;
либо через вынос общей основы;
либо через новую облегчённую реализацию.

Но итоговое пользовательское поведение должно быть каноническим и не хуже текущего чата в планах.

6. Приватные чаты
6.1. Новый раздел в ленте

Добавить 4-ю иконку на ленте:

Приватные чаты

Если есть непрочитанные сообщения:

на иконке раздела горит красный кружок.
6.2. Экран списка приватных чатов

Это полноценный экран, не шторка и не нижний sheet.

Заголовок:
Приватные чаты
Навигация:
только кнопка назад
крестика на экране списка нет
Пустое состояние:

Если приватных чатов нет, показывать текст:

«У вас пока нет приватных чатов»
Список:

Каждый чат находится в отдельном блоке/карточке.

Состав карточки:

аватар;
никнейм;
имя, если указано;
если имени нет — только аватар + ник;
красный кружок, если есть непрочитанные.

Ничего больше не добавлять:

без времени;
без превью последнего сообщения;
без вторичной информации.
Сортировка:
сначала чаты с непрочитанными;
внутри них — по последнему сообщению;
потом остальные — тоже по последнему сообщению.
Открытие:

Тап по карточке открывает конкретный приватный чат.

6.3. Экран конкретного приватного чата
Шапка:
аватар собеседника;
никнейм;
имя, если указано.
Навигация:
кнопка назад
крестик

Обе кнопки делают одно и то же:

закрывают текущий чат;
возвращают на экран списка приватных чатов.
6.4. Создание приватного чата
Где создаётся

На странице списка друзей.

На карточке друга нужно добавить кнопку:

Создать чат
Когда показывается кнопка

Кнопка показывается только если:

у пары ещё нет активного приватного чата;
нет блокировки.

Если чат уже есть:

кнопка не показывается.
Подтверждение

По нажатию показывать модалку:

«Создать чат с пользователем [ник/имя]?»

Кнопки:

Создать
Отмена
После подтверждения
чат создаётся сразу;
выполняется проброска прямо в созданный чат.
Важное правило

Пустой чат создавать можно.
Даже если ни одного сообщения потом не было:

чат остаётся в списке;
автоматически не удаляется.
6.5. Ограничения
один активный приватный чат на одну пару пользователей;
второй параллельный чат для той же пары запрещён;
повторное создание возможно только после удаления прошлого чата и только если нет блокировки.
6.6. Удаление чата

Внутри конкретного приватного чата должно быть действие:

Удалить чат
UX:

Показывается наша модалка подтверждения.

После подтверждения:

чат удаляется;
удаляется у обоих;
история удаляется;
возврат на список приватных чатов;
центральный тост:
«Чат удален»
После удаления

Если между пользователями нет блокировки:

позже чат можно создать заново.
6.7. Удалить чат и заблокировать пользователя

Внутри конкретного приватного чата должно быть действие:

Удалить чат и заблокировать пользователя
UX:

Показывается наша модалка подтверждения.

После подтверждения:

чат удаляется у обоих;
история удаляется;
дружба разрывается;
ставится блокировка;
возврат на список приватных чатов;
центральный тост:
«Пользователь заблокирован, чат удален»
6.8. Блокировка из списка друзей

На карточке друга в списке друзей добавить кнопку:

Заблокировать
Если приватного чата нет

После подтверждения:

дружба разрывается;
ставится блокировка;
центральный тост:
«Пользователь заблокирован»
Если приватный чат уже есть

Применяется жёсткий сценарий:

дружба разрывается;
ставится блокировка;
чат удаляется у обоих;
история удаляется;
центральный тост:
«Пользователь заблокирован»
6.9. Что означает блокировка

Блокировка:

не обязана скрывать пользователей друг от друга во всех местах интерфейса;
но запрещает любые дальнейшие взаимодействия.

После блокировки:

дружба разрывается;
чат удаляется;
история удаляется;
нельзя отправлять новые знаки внимания;
нельзя инициировать дружбу;
нельзя создать новый приватный чат.

Разблокировки в текущем ТЗ нет.

6.10. Если дружба удалена без блокировки

Если дружба исчезла без блокировки:

уже существующий приватный чат остаётся;
чат остаётся живым;
в него можно продолжать писать.

То есть:

дружба нужна для создания нового приватного чата;
но не обязательна для жизни уже созданного чата.
7. Механика сообщений приватного чата
7.1. Источник поведения

Все основные механики сообщений брать из уже существующего внутрипланового чата:

отправка;
realtime;
статусы;
непрочитанные;
обновление списков;
рендер;
клавиатура;
чтение;
реакции клиента на входящие сообщения.

Архитектор обязан сначала исследовать текущую реализацию и только затем переносить её в приватные чаты.

7.2. Первая версия типов сообщений

Поддержать:

текст;
emoji / смайлы.

Не поддерживать пока:

фото;
медиа;
вложения.
7.3. Поведение уведомлений
Когда приложение открыто
push не шлём;
на 4-й иконке Приватные чаты горит красный кружок;
на нужной карточке чата в списке тоже горит красный кружок.
Когда приложение свернуто или закрыто
приходит push;
тап по push открывает приложение на экране списка приватных чатов;
не в конкретный чат;
нужный чат определяется пользователем по красному кружку.
Текст push

Использовать:

«У вас новое сообщение в приватном чате. Откройте приложение чтоб посмотреть.»
8. Экран блокировок в профиле

Сейчас в профиле есть кнопки:

Настройки
Мои места
Рейтинг

Нужно изменить блок на:

Настройки
Блокировка
Мои места
Рейтинг
По тапу на «Блокировка»

Открывается экран:

заблокированные пользователи
Состав строки:
аватар;
ник;
дата блокировки.
Первая версия:
только просмотр;
без разблокировки;
без дополнительных действий.
9. Знаки внимания
9.1. Общая логика

Пользователь получает один случайный знак внимания в день.
Знак можно отправить только через специальный сценарий из списка участников плана в ленте.

9.2. Важная фиксация по ассетам знаков внимания

Стикеры / изображения знаков внимания уже находятся на сервере.

Их не нужно заново изобретать как отсутствующий контент и не нужно проектировать как “потом добавим”.
Нужно использовать уже существующие серверные ассеты.

По текущей фиксации пользователя:

ассеты лежат в Supabase Storage;
bucket: brand-media;
папка: signs of attention;
стикеры уже загружены и доступны как готовые изображения знаков внимания.

Следовательно, архитектору нужно:

исследовать текущую схему хранения и доступа к этим файлам;
не делать повторную загрузку тех же ассетов;
привязать типы знаков внимания к уже существующим серверным изображениям;
определить канонический способ выдачи клиенту ссылки/пути/идентификатора на нужный стикер;
убедиться, что клиент использует уже лежащие на сервере изображения.

То есть в рамках этого ТЗ:

графические ассеты знаков внимания уже есть;
задача состоит не в создании новых картинок, а в корректной серверной и клиентской интеграции уже загруженных стикеров.
9.3. Где показывается блок знаков

Это не сейф, а Коробка.

Размещение в профиле

Коробка должна находиться в профиле под блоком из 4 кнопок:

Настройки
Блокировка
Мои места
Рейтинг
Важное требование по вёрстке

Добавление Коробки может сместить текущие элементы профиля.
Перед внедрением нужно:

исследовать текущую вёрстку профиля;
понять, что там закреплено;
оценить возможные смещения;
внедрять аккуратно, без поломки экрана.
Коробка видна всегда

Даже если:

нет активного знака;
нет входящих;
нет накопленных принятых знаков.
9.4. Экран Коробки

Экран Коробки содержит 3 блока.

Блок 1. Накопления знаков

Показываются уже принятые знаки внимания с накопительным счётчиком:

какого типа сколько штук.

Если элементов много:

внутри блока появляется внутренний скролл.
Блок 2. На рассмотрении

Показываются входящие знаки, которые ещё не приняты и не отклонены.

Если пусто:

«Знаков внимания на рассмотрении нет»

Сортировка:

от старых к новым.

Если элементов много:

внутри блока появляется внутренний скролл.
Блок 3. Мой знак внимания

Показывается текущий активный знак, который пользователь может отправить.

Если знака нет:

«Сегодня свободных знаков не осталось. Ждите следующий знак.»
9.5. Выдача знаков

Нужно реализовать серверный механизм выдачи:

каждый день в 00:00;
выдаётся один знак;
знак выбирается случайным образом;
знак не копится;
знак не переносится;
если пользователь уже отдал сегодняшний знак, второго в этот день нет.
Количество типов

Пока типов знаков: 4.

Из них:

2 условно женских;
2 условно мужских.

Но:

отправлять можно любой знак любому пользователю;
ограничений по полу нет.
Важное требование

Типы знаков должны быть сопоставлены с уже лежащими на сервере стикерами.

9.6. Срок жизни знака

Срок жизни знака:

до следующей выдачи в 00:00.

Если знак был отправлен, но не принят до дедлайна:

он просто исчезает;
без уведомлений обеим сторонам.
9.7. Где можно отправить знак

Отправка возможна только в одном месте:

в ленте;
через видимый план;
внутри плана — через список участников;
по долгому зажатию на строке участника.

Нигде больше отправлять знак нельзя:

не по ID;
не из друзей;
не из профиля;
не из других экранов.
9.8. UX отправки знака

В списке участников плана по долгому зажатию открывать модалку:

«Вы действительно хотите отправить знак внимания пользователю [НИК]?»

Кнопки:

Отправить
Отмена

Если нажать Отправить:

знак сразу улетает;
у отправителя текущий знак исчезает;
у получателя он появляется в блоке На рассмотрении.

Перехода в Коробку для отправки нет.

9.9. Кому можно отправлять знак

Отправлять знак можно:

не друзьям;
друзьям;
пользователям, с которыми уже есть приватный чат.

Единственный серверный запрет:

наличие блокировки в любую сторону.
9.10. Повторная отправка знаков одному и тому же пользователю

Повторно отправлять знак одному и тому же пользователю можно, если:

наступил новый день;
у отправителя появился новый знак;
нет блокировки.
9.11. Принятие и отклонение знака

На входящем знаке в блоке На рассмотрении:

кнопка Принять
кнопка Отклонить
Если знак отклонён

Отправителю при следующем входе показать модалку:

«Пользователь [НИК] отклонил знак внимания»

Кнопка:

Закрыть
Если знак принят

У получателя:

знак фиксируется в блоке Накопления знаков;
увеличивается счётчик по соответствующему типу.

Отправителю при следующем входе показывается модалка.

Если получатель не друг

Модалка:

«Пользователь [НИК] принял знак внимания»

Кнопки:

Пригласить в друзья
Закрыть

Если нажать Пригласить в друзья:

запускается уже существующий флоу приглашения в друзья.

Если нажать Закрыть:

это право сгорает;
возможность была разовой.
Если получатель уже друг

Показывать модалку:

«Пользователь [НИК] принял знак внимания»

Кнопка:

только Закрыть

Кнопки приглашения в друзья в этом случае быть не должно.

10. Дружба
10.1. Использовать уже существующий флоу

Новый механизм дружбы не проектировать.
Использовать существующую систему:

отправка по ID;
отправка из списка участников плана;
принятие;
отклонение;
существующие серверные правила.
10.2. Связь со знаком внимания

Принятый знак внимания:

не делает пользователей друзьями автоматически;
только открывает разовую возможность отправить приглашение в друзья.
11. Очередь модальных событий

Если при входе в приложение накопилось несколько модальных событий:

ответы по знакам внимания;
ответы по дружбе;
аналогичные события;

показывать их:

очередью;
по одной модалке;
от старых к новым.
12. Серверные требования

Архитектор должен спроектировать серверную модель так, чтобы она поддерживала:

12.1. Приватные чаты
сущность приватного чата 1-на-1;

очередью;
с контролем статуса показа/потребления.

13.4. Изменения в списке друзей

На карточке друга:

кнопка Создать чат
кнопка Заблокировать

С учётом серверных условий показа.

13.5. Изменения в списке участников плана

В списке участников плана:

по долгому зажатию по строке участника —
открывать модалку отправки знака внимания.
13.6. Работа с ассетами знаков внимания

Клиент должен:

не хранить “свои” альтернативные локальные знаки как источник истины;
получать от сервера тип/идентификатор/путь знака;
рендерить уже существующий серверный стикер;
корректно показывать его:
в блоке “Мой знак внимания”;
в блоке “На рассмотрении”;
в “Накоплениях”.

15. Что не входит в первую версию

Не делать в этой итерации:

фото в приватных чатах;
медиа и файлы;
разблокировку;
дополнительные способы отправки знака;
отдельный сложный профиль собеседника;
избыточные вторичные сценарии вне описанного контракта.

17. Короткая постановка для архитектора

Нужно реализовать приватные чаты 1-на-1, систему знаков внимания, экран блокировок и связанный флоу перехода от принятого знака внимания к дружбе.
Базовую механику сообщений, realtime, статусов и UX нужно исследовать и перенести из уже существующего внутрипланового чата.
При этом приватные чаты — это отдельный раздел приложения со своим экраном списка, своей навигацией и своим контрактом уведомлений.
Новая логика должна быть реализована строго в модели server-first.
Отдельно важно: стикеры знаков внимания уже загружены на сервер в storage, поэтому их не нужно создавать заново — нужно спроектировать корректную серверную и клиентскую интеграцию уже существующих ассетов.
